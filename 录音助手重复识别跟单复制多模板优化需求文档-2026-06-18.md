## 1. 本次需求目标

本次只做三类优化。

第一类：网页录音重复识别防护。

用户在网页录音监听环节，如果当前登录账号自己已经识别过同一段录音，插件要提醒“已识别过”，并优先引导用户打开已有记录，避免重复上传和重复 DashScope 转写。

第二类：红娘/招聘跟单信息一键复制。

红娘部门和招聘部门需要把 AI 整理出的跟单内容复制粘贴到业务系统里。跟单复制格式必须严格对齐业务系统字段；字段里没有识别到内容时，复制出来留空。

第三类：普通会议纪要增加多种总结模板。

当前会议纪要只有一个默认模板。后续需要增加类似 Plaud 的多模板选择能力：推理总结、会议总结全面专家、会议秘书、智能摘要、电话讨论。用户不选择时，默认仍使用当前推理总结。

## 2. 明确不做的范围

1. 不做跨员工查重。
2. 不做跨部门查重。
3. 不做管理员/老板全局查重。
4. 不做手动上传文件 hash 查重。第一期只做网页监听录音查重。
5. 不把“电话讨论”当成跟单模板。电话讨论只是普通会议总结模板之一。
6. 不改变当前 R2、DashScope、总结模型池、登录鉴权、权限体系的核心架构。
7. 不改变普通员工查看权限。
8. 不因新增模板影响原有默认会议纪要体验。
9. 不让同一录音换模板时重新转写。

## 4. 功能一：网页录音重复识别提醒

### 4.1 用户规则

重复识别只按当前登录用户自己判断。

场景：

1. 用户 A 登录插件。
2. 用户 A 在 Plaud 或其他网页监听到一条录音。
3. 系统根据候选录音 URL 生成稳定指纹。
4. 后端只查询用户 A 自己的记录。
5. 如果用户 A 以前识别过同一段录音，前端提示已识别过。

不允许：

1. 不查询其他员工记录。
2. 不查询同部门其他人记录。
3. 不因为当前账号是管理员或老板，就扩大到全公司查重。
4. 不在普通员工界面暴露别人是否识别过某条录音。

### 4.2 稳定指纹规则

新增后端工具文件建议：

`/Users/lixin/AI code/voice to word/server/lib/media-fingerprint.js`

导出：

```js
normalizeMediaUrlForFingerprint(url)
mediaUrlFingerprint(url)
isVolatileMediaQueryParam(key)
```

`normalizeMediaUrlForFingerprint(url)` 规则：

1. 保留协议。
2. 保留域名。
3. 保留路径。
4. 删除 hash。
5. 删除临时鉴权 query 参数。
6. 保留能区分录音的业务 query 参数。
7. 对剩余 query 参数排序。
8. 返回稳定字符串。

需要删除的临时参数包括：

```text
token
access_token
auth
authorization
signature
sig
expires
expires_at
expiration
policy
credential
date
security-token
x-amz-*
x-oss-*
response-*
_*
```

需要保留的参数示例：

```text
id
recordId
recordingId
fileId
object
key
mediaId
audioId
```

测试样例：

```text
https://a.com/audio?id=1&token=aaa
https://a.com/audio?token=bbb&id=1
```

这两个应生成相同指纹。

```text
https://a.com/audio?id=1
https://a.com/audio?id=2
```

这两个应生成不同指纹。

### 4.3 兼容老数据

老数据里 `source_media_url_hash` 可能是直接对原始 URL 做 sha256。

为了兼容老数据：

1. 新记录写入 normalized hash。
2. 查重时同时计算：
   - normalized hash
   - raw hash
3. 查询时匹配：
   - `source_media_url_hash === normalizedHash`
   - 或 `source_media_url_hash === rawHash`

历史限制：

如果旧记录只保存了一个每次都会变化的签名 URL hash，后续可能匹配不上。这是可接受限制，不要为了这件事迁移所有真实数据。

### 4.4 新增查重 API

新增接口：

`POST /api/records/check-duplicate`

请求体：

```json
{
  "candidateUrl": "https://example.com/audio?id=123&token=xxx",
  "sourcePageUrl": "https://example.com/detail/123",
  "sourcePageTitle": "2026-06-18 外呼录音"
}
```

未重复返回：

```json
{
  "duplicate": false
}
```

已重复返回：

```json
{
  "duplicate": true,
  "record": {
    "id": "record-id",
    "title": "客户电话沟通",
    "status": "completed",
    "createdAt": "2026-06-18T10:00:00.000Z",
    "updatedAt": "2026-06-18T10:20:00.000Z",
    "templateType": "meeting_minutes",
    "followupType": "none",
    "hasTranscript": true,
    "hasSummary": true,
    "durationSeconds": 3600,
    "fileSize": 12345678
  }
}
```

查询要求：

1. 必须登录。
2. 只查当前登录用户自己的记录：
   `record.owner_employee_id === auth.employee.id`
3. 忽略永久删除或软删除记录：
   `deleted_at` 有值时不参与重复提醒。
4. 归档记录仍参与重复提醒，因为它仍然属于用户自己的历史记录。
5. 老记录没有 `source_media_url_hash` 时，不参与去重。

哪些状态算已识别过：

```text
uploaded
transcribing
summarizing
transcribed
completed
failed
```

不同状态的提示：

1. `completed`：这条录音已识别完成。
2. `transcribed`：这条录音已转写，可直接重新生成总结。
3. `uploaded` / `transcribing` / `summarizing`：这条录音已有任务处理中。
4. `failed`：这条录音之前识别失败，可打开记录重试。

### 4.5 创建记录保护

为避免前端漏查，`POST /api/records` 也要增加保护。

请求体新增：

```json
{
  "forceDuplicate": false
}
```

默认：

`forceDuplicate = false`

触发保护条件：

1. `sourceType = web_capture`
2. 请求体有 `candidateUrl`
3. 当前登录用户自己已有同样 `source_media_url_hash`
4. `forceDuplicate !== true`

返回 HTTP 状态：

`409`

返回体：

```json
{
  "error": "这条录音已识别过",
  "duplicate": true,
  "record": {
    "id": "record-id",
    "title": "客户电话沟通",
    "status": "completed",
    "createdAt": "2026-06-18T10:00:00.000Z"
  }
}
```

前端收到 409 时：

1. 不继续上传。
2. 不继续转写。
3. 提示用户已识别过。
4. 提供“打开已有记录”。

用户明确点击“仍要重新识别”时：

1. 前端弹出二次确认。
2. 用户确认后传 `forceDuplicate: true`。
3. 后端允许创建新记录。

手动上传：

不走这个保护。

### 4.7 弹出提醒规则

用户要求识别到重复录音时能提醒。

建议不要使用浏览器系统弹窗。使用页面内醒目提示更稳定。

实现方式：

1. 候选卡片内显示“已识别过”。
2. 页面顶部状态提示：
   `这条录音已识别过，已为你找到历史记录。`
3. 点击“仍要重新识别”时，弹二次确认。

二次确认文案：

```text
这条录音已经识别过，继续会重新上传并重新转写，可能产生额外费用。是否继续？
```

按钮：

1. `取消`
2. `继续重新识别`

默认建议选中或突出 `取消`。

### 4.8 同录音换模板规则

用户已确认：

同一录音选择不同模板，直接复用逐字稿即可。

正确流程：

1. 监听候选发现重复。
2. 用户点击 `打开已有记录`。
3. 进入记录详情页。
4. 用户在“总结模板”下拉框切换模板。
5. 点击 `重新生成`。
6. 前端调用：
   `POST /api/records/:id/summarize`
7. 后端复用已有 transcript。
8. 只重新生成 summary。

禁止：

1. 禁止重新上传录音。
2. 禁止创建新的 audio_record。
3. 禁止重新调用 DashScope 转写。
4. 禁止调用 `POST /api/records/:id/transcribe`。

验收：

1. 重新生成总结时，不新增新的 DashScope `asr_task_id`。
2. `record_processing_events` 只新增总结相关事件。
3. 历史列表仍只有一条原录音。

### 5.2 复制总规则

1. 复制内容必须是纯文本。
2. 不使用 Markdown。
3. 不出现 `#`、`##`、`*`、代码块。

| 待补充信息 | `fields.pendingInfo` |

阶段中文名称沿用现有招聘五阶段：

| stage | 中文 |
| --- | --- |
| `initial_effective_followup` | 初期有效跟进 |
| `mid_effective_followup` | 中期有效跟进 |
| `no_hiring_followup` | 暂不招人有效跟进 |
| `mid_late_effective_followup` | 中后期有效跟进 |
| `late_effective_followup` | 后期有效跟进 |

### 5.5 通用客户跟单复制格式

如果 `followupType = general_customer`，复制：

```text
【客户状态】：
【客户/企业名称】：
【沟通关键信息】：
【客户需求】：
【客户顾虑】：
【下一步动作】：
```

### 5.6 编辑后的复制规则

如果用户编辑过跟单，复制必须优先使用已保存的编辑内容。

优先级：

1. 已保存的 `followup_forms`
2. `fields_json`
3. `followup_markdown` 仅作为兜底

如果用户正在编辑抽屉里修改内容但尚未保存，点击复制时建议提示：

`当前跟单有未保存修改，请先保存后复制。`

第一期也可以不判断未保存差异，但不能复制旧内容后提示成功而误导用户。

### 12.2 换模板复用逐字稿

2. 切换总结模板。
3. 点击重新生成。
4. 只进入 summarizing。
5. 不重新上传录音。
6. 不重新转写。
7. 历史记录仍只有一条。
8. 新总结使用新模板格式。

## 13. 给执行 AI 的注意事项

1. 不要先大改 UI 架构。
2. 不要把“重复识别”做成全公司查重。
3. 不要为了去重暴露别人的记录。
4. 不要重新设计 DashScope 转写链路。
5. 不要把“电话讨论”混进红娘/招聘跟单。
6. 不要把空字段填成“待核对”，复制格式里空就空。
7. 不要让同录音换模板重新消耗 ASR。
8. 不要修改真实密钥、真实数据和局域网地址。
9. 任何新增接口都必须有测试。
10. 前端扩展变更后必须升版本并打包。
