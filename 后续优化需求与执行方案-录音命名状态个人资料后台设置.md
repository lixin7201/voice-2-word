# 大宜宾录音助手后续优化需求与执行方案

更新时间：2026-06-17

本文只做需求和执行方案，不改业务代码。下一窗口可直接按本文件分阶段实现。

## 1. 本次用户反馈

1. Plaud 网页录音点击识别后，录音标题是文件哈希名，用户无法重命名；希望用户可手动改名。如果用户不改名，希望录音转文字和总结完成后，由 LLM 根据录音主题自动生成简短标题。
2. 16 分钟录音等待十几分钟仍然看起来没识别出来，而 AI 大宜宾里已经能很快出结果。需要确认真实链路是否可用，并让界面清楚显示处理进度。
3. 当前没有改密码入口；希望点击右上角头像进入个人资料，可修改密码、头像、一句话自我介绍等。自我介绍要能作为 LLM 生成记录时的用户背景，让结果更精准。
4. 后台配置页看起来不能修改各种参数；希望新增清晰的设置功能，仅管理员可设置，并确认管理员修改参数后，已安装插件的员工是否能同步使用最新配置。

## 2. 当前实现检查结论

### 2.1 16 分钟 Plaud 录音真实状态

从 `data/dev-db.json` 检查到用户截图中的记录：

- 记录 ID：`b3208418-53f1-465b-a447-25cc2cf5a06f`
- 原始标题：`0ba0d1558bd6dcd2723bcab64a3c9799.mp3`
- 来源：`web_capture`
- 来源页面：`https://web.plaud.cn`
- 创建时间：`2026-06-17 09:11:08`
- 完成时间：`2026-06-17 09:14:15`
- 音频时长：`1009` 秒，约 `16 分 49 秒`
- 当前状态：`completed`
- ASR：`dashscope`
- ASR task id：`03fdd58a-0f5d-4574-8f50-2a6106ac296e`
- 逐字稿长度：`3448` 字
- 总结模型：`easyai / gpt-5.5`
- 总结已生成，长度约 `484` 字

结论：真实链路是跑通的，16 分钟录音实际约 3 分钟完成。用户看到“已上传 / 暂无总结”的原因是前端在上传后立刻打开详情页，但后端转写和总结是异步完成的；详情页没有自动轮询刷新，所以页面停留在旧状态。

### 2.2 标题能力现状

当前能力：

- 手动上传页有“标题”输入框。
- 网页捕获候选没有“标题”输入框。
- 详情页没有重命名入口。
- 后端 `POST /api/records` 接受初始 `title`。
- 后端没有 `PATCH /api/records/:id` 之类的记录标题修改接口。
- LLM 总结返回字段目前不包含标题建议。

缺口：

- 用户不能在 Plaud 候选上传前重命名。
- 用户不能在详情页重命名。
- 哈希文件名会长期保留，影响历史检索。
- LLM 总结完成后不会自动生成主题标题。
- 没有区分“用户手动标题”和“系统/AI 自动标题”，后续容易误覆盖用户修改。

### 2.3 个人资料和改密码现状

当前能力：

- 后端已有 `POST /api/auth/change-password`。
- 接口校验旧密码，且新密码至少 6 位。
- 右上角头像按钮当前只绑定退出登录。
- `employees` 当前字段只有工号、登录名、花名、密码哈希、角色、状态、最后登录时间。
- 没有头像、自我介绍、AI 生成偏好字段。

缺口：

- 没有改密码 UI。
- 没有个人资料页或弹窗。
- 没有头像上传/选择。
- 没有自我介绍字段。
- LLM prompt 没有使用用户背景。

### 2.4 后台配置现状

当前能力：

- 只有 `admin` 和 `boss` 可以访问 `/api/admin/settings`。
- 配置保存在后端 `system_settings`，不会写进员工浏览器插件。
- `resolveRuntimeConfig(config, store)` 每个请求都会重新从数据库解析配置。
- 因此管理员修改 DashScope、R2、EasyAI、Kimi 等后端参数后，员工下一次上传/转写会自动使用新参数。
- 密钥字段不会明文回传给前端，只返回空值、已配置状态和脱敏值。
- 空提交密钥不会覆盖旧密钥，除非勾选清空。

缺口：

- 配置页是一个长表单，顶部看不到“保存”，用户容易误以为不能改。
- 密钥字段“已保存：sk-...xxx”显示在 placeholder，交互不直观。
- 没有“保持现有 / 替换 / 清空”的明确动作。
- 没有单项保存，也没有测试按钮。
- 没有显示每个参数的来源：后台保存、环境变量、默认值。
- 没有配置版本号或变更时间，员工端是否已使用最新配置不可见。
- 如果只改 ASR/LLM/R2 参数，员工插件无需重装；但如果改 `publicBaseUrl` 或后端真实地址，员工浏览器本地保存的 `apiBaseUrl` 可能仍指向旧地址，需要明确提示。

## 3. 目标和非目标

### 3.1 目标

1. 录音标题可控：
   - 用户上传/识别前可填标题。
   - 详情页可随时重命名。
   - 用户没改名时，LLM 自动生成简短标题。
   - 用户手动改过的标题不被 AI 覆盖。

2. 处理状态可见：
   - 上传后展示“已上传 / 转写中 / 总结中 / 已完成 / 失败”。
   - 详情页和历史页自动刷新进行中的记录。
   - 长录音有明确等待提示和后台任务信息。

3. 个人资料可维护：
   - 点击右上角头像进入“个人资料”。
   - 可修改头像、自我介绍、AI 生成偏好、密码。
   - 自我介绍进入 LLM 总结上下文。

4. 管理员设置中心可用：
   - 设置页改成清晰的“设置中心”。
   - 支持分组编辑、单项测试、保存反馈、审计日志。
   - 明确员工端同步规则。

### 3.2 非目标

1. 不在 Chrome 扩展内保存 DashScope、R2、EasyAI、Kimi 密钥。
2. 不让普通员工修改全局配置。
3. 不让 LLM 覆盖用户手动编辑过的标题、跟单或资料。
4. 不把头像、密钥、个人资料明文暴露到无权限接口。
5. 不在本阶段做完整 Supabase 上线迁移，但必须同步更新 Supabase schema。

## 4. 详细需求

### 4.1 录音重命名和自动命名

#### 4.1.1 用户故事

- 作为员工，我在 Plaud 页面识别录音时，可以先给录音起一个看得懂的名字。
- 作为员工，我识别完成后发现标题不准确，可以在详情页直接修改。
- 作为员工，如果我懒得改名，系统应该自动把 `0ba0d155...mp3` 改成类似“录音助手功能讨论”这样的短标题。
- 作为管理员或老板，我查看历史记录时，能通过清晰标题快速判断录音内容。

#### 4.1.2 数据字段

在 `audio_records` 增加：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `title_source` | text | `filename` / `manual` / `ai` / `page`，标题来源 |
| `title_locked` | boolean | 用户手动改名后为 `true`，AI 不得覆盖 |
| `ai_title` | text | LLM 生成的标题建议 |
| `title_updated_at` | timestamp | 标题最后更新时间 |

兼容旧数据：

- 如果现有 `title` 等于 `original_file_name` 或像哈希文件名，迁移时设为 `title_source='filename'`、`title_locked=false`。
- 如果现有 `title` 是人工可读标题，迁移时设为 `title_source='manual'`、`title_locked=true`。

哈希文件名判断建议：

- 去掉扩展名后长度大于等于 24。
- 主要由小写字母和数字组成。
- 没有中文、空格或业务词。

#### 4.1.3 后端 API

新增：

`PATCH /api/records/:id`

请求：

```json
{
  "title": "录音助手功能讨论",
  "titleSource": "manual"
}
```

权限：

- 记录所有者可以改自己的标题。
- `admin` / `boss` 可以改所有标题。
- 部门领导是否可改部门成员标题：建议第一阶段只允许查看，不允许改，避免责任混乱。

返回：

```json
{
  "record": {}
}
```

行为：

- `title` trim 后不能为空。
- 最大长度建议 60 字。
- 保存后 `title_locked=true`、`title_source='manual'`。
- 写入审计日志：`rename_record`。

#### 4.1.4 前端 UI

网页捕获候选列表：

- 每个候选卡片增加“录音标题”输入框。
- 默认值优先级：
  1. `candidate.pageTitle`
  2. `candidate.name`
  3. `网页录音`
- 点击“开始识别”时，将输入框标题传入 `createRecord()`。

手动上传页：

- 当前已有标题输入框，保留。
- 如果用户不填，默认先用文件名，但 `title_source='filename'`、`title_locked=false`。

详情页：

- 标题旁边增加编辑按钮。
- 点击后变成输入框 + 保存 / 取消。
- 保存成功后刷新当前详情和历史列表。

历史列表：

- 标题下增加来源提示：
  - 人工命名
  - AI 命名
  - 文件名
- 文件名来源但已完成的记录，如果 AI 标题生成失败，显示一个轻提示：“可点击详情页改名”。

#### 4.1.5 LLM 自动标题

修改 `server/lib/templates.js` 的 prompt 合同，增加字段：

```text
必须返回 JSON 字段：
summaryMarkdown, overviewCard, structuredJson, followupMarkdown,
followupFields, followupStage, suggestedTag, statusLabel,
customerName, companyName, titleSuggestion
```

`titleSuggestion` 规则：

- 中文为主。
- 8 到 18 个汉字左右，最长不超过 30 个中文字符。
- 不要出现“会议纪要”“录音”“转写”等泛词，除非它就是主题。
- 不要包含日期。
- 不要包含文件扩展名。
- 不要包含明显不确定的人名，除非逐字稿明确。
- 例：`录音助手功能讨论`、`招聘客户跟进沟通`、`APP 数据打通讨论`。

修改 `server/lib/llm.js`：

- 解析 `titleSuggestion`。
- 返回给 `summarizeExistingRecord()`。

修改 `summarizeExistingRecord()`：

- 如果 `record.title_locked !== true` 且 `titleSuggestion` 有效：
  - 更新 `audio_records.title = titleSuggestion`
  - `ai_title = titleSuggestion`
  - `title_source = 'ai'`
  - `title_updated_at = now`
- 如果用户已手动改名，只保存 `ai_title` 作为建议，不覆盖 `title`。

本地模板兜底：

- `buildLocalSummary()` 也返回 `titleSuggestion`。
- 简单规则：取前 20 字关键主题；如果提取不到则不改。

### 4.2 转写状态、长录音体验和自动刷新

#### 4.2.1 当前问题

后端真实处理流程已经可用，但前端没有自动刷新。用户上传后看到“已上传 / 暂无总结”，误以为没识别。

#### 4.2.2 数据字段

在 `audio_records` 增加：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `processing_started_at` | timestamp | 后端开始处理时间 |
| `transcribe_started_at` | timestamp | 开始转写时间 |
| `summarize_started_at` | timestamp | 开始总结时间 |
| `last_progress_at` | timestamp | 最近一次进度更新时间 |
| `asr_task_id` | text | DashScope task id，提交后立即写入 |
| `processing_attempts` | integer | 处理尝试次数 |

可选新增表 `record_processing_events`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | 主键 |
| `audio_record_id` | uuid | 记录 ID |
| `phase` | text | `uploaded/transcribing/summarizing/completed/failed` |
| `message` | text | 面向用户或管理员的进度说明 |
| `metadata_json` | json | task id、耗时、模型名等 |
| `created_at` | timestamp | 创建时间 |

第一阶段可以先只加字段，不建 events 表；但如果要做“诊断面板”，建议建 events 表。

#### 4.2.3 后端行为

`processUploadedRecord()`：

1. 上传 R2 前：
   - `status='uploaded'`
   - `processing_started_at=now`
   - `last_progress_at=now`
2. 提交 DashScope 成功后：
   - `status='transcribing'`
   - `asr_task_id=taskId`
   - `transcribe_started_at=now`
   - `last_progress_at=now`
3. 下载转写结果后：
   - 写入 `transcripts`
   - `duration_seconds=...`
   - `status='summarizing'`
   - `summarize_started_at=now`
4. 总结完成后：
   - 写入 `summaries` 和 `followup_forms`
   - 更新 AI 标题
   - `status='completed'`
   - `completed_at=now`
5. 失败：
   - `status='failed'`
   - `error_message=...`
   - `last_progress_at=now`

重要修正：

- 当前 `transcribeWithDashScope()` 只有完成后才返回 taskId。应拆成：
  - `submitDashScopeTask(config, audioUrl)` 返回 taskId。
  - `waitForDashScopeTask(...)` 轮询。
  - 这样可把 taskId 早写入数据库，方便排查。

#### 4.2.4 前端自动刷新

详情页：

- 当 `record.status` 属于 `uploaded/transcribing/summarizing` 时，开启轮询。
- 每 3 秒请求 `/api/records/:id`。
- 状态变为 `completed/failed` 后停止轮询。
- 离开详情页时停止轮询。

历史页：

- 当列表里存在非终态记录时，每 5 秒刷新一次历史。
- 只在当前 view 是 `home/history/detail` 时刷新，避免设置页和员工页被打断。

状态文案：

| 状态 | 展示 |
| --- | --- |
| `created` | 已创建，等待上传 |
| `uploading` | 正在上传 |
| `uploaded` | 已上传，等待转写 |
| `transcribing` | 正在转文字，长录音可能需要几分钟 |
| `summarizing` | 正在生成总结和跟单 |
| `completed` | 已完成 |
| `failed` | 处理失败 |

详情页进度区：

- 显示文件名、时长、文件大小。
- 如果有 `asr_task_id`，显示“转写任务已提交”。
- 如果超过 5 分钟仍未完成，显示“长录音仍在处理中，可稍后回来查看”。
- 如果失败，提供“重新转写”按钮。

#### 4.2.5 验收标准

1. 上传 16 分钟 Plaud MP3 后，详情页无需手动刷新，自动从“已上传”变成“转写中 / 总结中 / 已完成”。
2. 完成后 5 秒内详情页出现总结和逐字稿。
3. 历史页状态自动更新为“完成”。
4. 失败时显示真实错误，并且“重新转写”可用。
5. 服务重启后，已完成记录仍能打开；失败记录可重新触发。

### 4.3 个人资料、头像、改密码、自我介绍

#### 4.3.5 LLM 使用用户资料

修改总结入口：

- `summarizeExistingRecord(record, store, config)` 获取 owner employee。
- 构造 profile context：

```text
录音所属员工：
- 花名：离心
- 部门：运营部
- 一句话介绍：负责 AI 工具和内部效率系统
- AI 生成偏好：我更喜欢清晰、可执行、偏产品化的会议纪要。
```

修改 `buildSummaryPrompt(record, transcriptText, context)`：

- system prompt 仍要求只依据逐字稿，不编造事实。
- user prompt 增加“员工背景仅用于选择表达重点和字段优先级，不得把背景当作录音事实写入纪要。”

验收：

## 11. 二期修复与 AI 大宜宾录音总结复刻需求

更新时间：2026-06-17

本节基于最新使用反馈追加，作为下一轮代码修改的执行依据。核心目标不是新增花哨功能，而是把当前插件变成“普通员工也能稳定点得动、看得懂、用得上”的录音工作台。

### 11.1 最新问题归纳

1. 录音已经识别到后，点击修改名称进入编辑态，但保存和取消都没有反应；Chrome 错误页显示 `Form submission canceled because the form is not connected`。
2. 配置页密钥的“替换”无法稳定选择，导致管理员无法确认配置是否真的修改并生效。
3. 当前每条录音默认生成会议纪要和跟单信息，会议场景也生成跟单，造成 token 浪费和内容污染。
4. 用户希望纪要类型和跟单信息可以自由选择，并且选择后下次默认沿用。
5. 录音详情页需要显示录音播放器，逐字稿可以对照录音，点击时间戳或段落能跳转播放。
6. 会议纪要需要 1:1 复刻 AI 大宜宾录音总结相关功能和界面：总结卡片、文字重点总结、重点标注、思维导图。
7. 下载范围按四种格式验收：Markdown、TXT、DOCX、PDF。

### 11.2 交互稳定性硬要求

所有会保存数据的控件都必须满足：

1. 不依赖容易被重绘打断的裸表单提交。
2. 保存按钮必须是明确动作按钮，点击后只触发一次保存。
3. 取消按钮必须立即退出编辑态，不请求后端。
4. 保存中按钮禁用，并显示清晰状态。
5. 保存成功要刷新当前数据并显示轻提示。
6. 保存失败要保留用户输入并显示错误，不得直接清空或卡住。

第一阶段必须修复：

1. 录音重命名保存/取消。
2. 配置密钥“保持现有 / 替换 / 清空”选择。
3. 配置保存。
4. 后续如发现个人资料、备注、跟单编辑存在同类表单断开问题，也按同一模式修复。

### 11.3 纪要和跟单选择规则

录音处理拆成两个概念：

1. `templateType`：决定总结/纪要模板。
2. `followupType`：决定是否生成跟单，以及生成哪类跟单。

`templateType` 可选：

| 值 | 展示文案 | 说明 |
| --- | --- | --- |
| `meeting_minutes` | 会议纪要 | 普通会议、内部讨论、复盘会议默认使用 |
| `business_review` | 业务复盘 | 业务问题、流程优化、经营复盘使用 |
| `customer_follow_up` | 客户跟进总结 | 面向普通客户沟通的总结模板 |
| `matchmaker_profile` | 红娘客户画像 | 红娘部门客户画像总结模板 |
| `recruitment_followup` | 招聘客户沟通总结 | 招聘部门客户沟通总结模板 |

`followupType` 可选：

| 值 | 展示文案 | 说明 |
| --- | --- | --- |
| `none` | 不生成跟单 | 默认会议纪要、业务复盘使用 |
| `general_customer` | 通用客户跟单 | 用户明确选择客户跟进时生成 |
| `matchmaker` | 红娘跟单/画像字段 | 红娘部门或用户明确选择时生成 |
| `recruitment` | 招聘跟单 | 招聘部门或用户明确选择时生成 |

默认规则：

1. 用户上一次选择优先，保存在浏览器本地。
2. 没有历史选择时，按员工部门给推荐默认：
   - 红娘部门：`templateType=matchmaker_profile`，`followupType=matchmaker`。
   - 招聘部：`templateType=recruitment_followup`，`followupType=recruitment`。
   - 其他部门：`templateType=meeting_minutes`，`followupType=none`。
3. 用户可以随时改成“不生成跟单”。
4. 当 `followupType=none` 时：
   - LLM prompt 不要求输出跟单字段。
   - 后端不写入新的 `followup_forms`。
   - 前端不显示跟单 tab。
   - 导出完整记录不包含“暂无跟单”。

### 11.4 AI 大宜宾录音总结工作台复刻范围

复刻方式：不引入 React 构建链，在当前 `sidepanel.js + sidepanel.css` 内移植 AI 大宜宾的工作台结构、样式和数据能力。

必须复刻的功能：

1. 顶部显示录音标题、处理状态、下载入口。
2. 左侧显示录音播放器。
3. 左侧显示可搜索逐字稿列表。

### 11.5 后端数据与接口要求

新增或调整：

1. `audio_records.followup_type`
   - 本地 JSON 数据和 Supabase schema 同步增加。
   - 旧数据默认按 `template_type` 迁移，会议纪要和业务复盘为 `none`。

2. `POST /api/records`
   - 接收 `followupType`。
   - 未传时使用部门推荐默认。

3. `POST /api/records/:id/summarize`
   - 接收 `templateType` 和 `followupType`。
   - 重新生成时按最新选择更新记录。

4. `GET /api/records/:id`
   - 返回 `followupType`。
   - 返回 `audioUrl`，用于前端播放器。

5. `GET /api/records/:id/audio`
   - 当前用户有查看权限才可访问。
   - 本地文件支持浏览器播放。
   - R2 文件使用临时链接跳转或转发。

### 11.6 导出要求

下载范围固定为四种格式：

1. Markdown。
2. TXT。
3. DOCX。
4. PDF。

导出目标：

1. 逐字稿导出：原始/整理后的逐字稿内容。
2. 总结导出：总结卡片、文字重点总结、思维导图。
3. 完整记录导出：总结、逐字稿，只有存在跟单时才追加跟单。

验收：

1. `followupType=none` 的记录，导出文件中不能出现“跟单”或“暂无跟单”。
2. 总结导出必须包含卡片和思维导图对应内容。
3. 四种格式接口均返回可下载文件。
