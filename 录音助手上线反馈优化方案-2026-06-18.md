# 大宜宾录音助手上线反馈优化方案

更新时间：2026-06-18

本文只做排查结论和可执行优化方案，不直接修改业务代码。目标是让下一窗口或另一位 AI 能按本文逐项落地、测试、上线。

## 0. 本次反馈

1. 办公室已有同事在用。有人反馈：录音转文字成功，页面提示“总结已完成”，但生成的总结几乎没有内容。
2. 后续新增功能时，希望像 `/Users/lixin/AI code/ztools` 一样，在局域网内显示更新提示，并尽量自动更新到新版插件。
3. 管理员账号/练团长账号能看到所有人的录音，但历史列表混在一起，太杂。希望按人展开查看。

## 1. 结论摘要

### 1.1 “总结完成但没有内容”的真实原因

这次样本不是 DOCX 文件损坏，也不是总结表完全没写入。真实情况是：

- 记录 ID：`29cf42a6-6c7a-417f-a746-577162b8c666`
- 创建时间：`2026-06-18T03:36:20.498Z`
- 状态：`completed`
- 逐字稿长度：约 `1308` 字
- 总结模型：`local-template`
- 真实模型错误：
  - EasyAI GPT-5.5：模型请求超时
  - AI 大宜宾 sub2api GPT-5.5：模型请求超时
  - Kimi K2.6：配额用尽
- 最终总结长度：约 `110` 字，主要是“会议背景：不能贷了，两三天”和多处“待核对”

也就是说：三家真实总结模型都失败后，后端降级生成了本地模板；但后端仍把记录标成 `completed`，前端也把它当正常总结展示和导出，于是同事看到的是“完成了，但内容空”。

### 1.2 当前代码关键点

当前总结链路：

- `server/lib/llm.js`
  - `generateSummary()` 按模型池优先级调用真实模型。
  - 如果所有模型都失败，会返回 `buildLocalSummary()` 的本地模板，并写入 `modelProvider = local-template`。
  - 当前本地模板返回后没有质量等级。

- `server/app.js`
  - `summarizeExistingRecord()` 无论真实模型成功还是本地模板兜底，都会写入 `summaries`，然后把 `audio_records.status` 更新为 `completed`。
  - `record_processing_events` 写入“总结已生成”。

- `sidepanel.js`
  - `hasSummaryContent()` 只判断 `summary_markdown` / `overview_card_json` / `mind_map_json` 是否存在。
  - 只要存在模板壳，就会被认为“有总结”。
  - 历史列表是扁平列表，管理员和老板看到所有可见记录后按时间混排。

### 1.3 应优先修的事情

第一优先级不是“让本地模板写得更漂亮”，而是：

1. 不允许真实模型全失败后的兜底模板被标成正常完成。
2. 前端必须区分：
   - 转写完成
   - AI 总结完成
   - AI 总结失败但保留逐字稿
   - 总结内容质量不足
3. 管理员历史页默认按人分组，避免全员录音混在一起。
4. R2 上传不能因为文件名带空格、括号等字符而失败；上传失败时要给同事可理解提示，给管理员可定位诊断。
5. 插件更新可以做“局域网版本提示 + 下载/重载引导”；若要接近 ztools 的自动更新，需要改成打包 CRX 或企业策略安装，不能指望已加载的解压扩展自己静默覆盖文件。

## 2. 设计原则

1. 上线工具先稳再快：不能因为总结失败导致逐字稿丢失，也不能把失败伪装成成功。
2. 状态要说人话：同事看到的文案应该是“转写已完成，AI 总结失败，可重试”，而不是“总结已完成”。
3. 后台配置问题要暴露给管理员：模型超时、配额用尽、未配置 Key，都要在管理员可见位置提示。
4. 改动要小步上线：先做质量闸门和历史分组，不先大改 ASR 或 UI 架构。
5. Chrome 解压扩展不能像 Electron 应用那样完全静默自更新；要明确不同发布模式的能力边界。

## 3. 优化一：总结完成但内容空

### 3.1 目标

用户侧目标：

- 真实模型生成成功时，状态显示“完成”，可复制、下载总结。
- 真实模型失败但逐字稿成功时，状态显示“已转写，AI 总结待重试”。
- 如果只有本地模板兜底，页面要明确标注“不建议直接使用”，并提供一键重试。
- 导出的 DOCX/PDF 不再悄悄导出一个看似正式、实际空壳的总结。

管理员侧目标：

- 能看到是哪一个模型失败、失败原因是什么。
- 能在配置页看到最近一次调用状态，并测试/调整模型池。
- 能从记录详情复制诊断给技术人员。

### 3.2 数据字段调整

在 `summaries` 增加以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `quality_status` | text | `ai_ok` / `fallback_template` / `low_information` / `invalid` |
| `quality_reason` | text | 面向用户和管理员的简短原因 |
| `input_transcript_chars` | integer | 生成总结时使用的逐字稿字数 |
| `summary_chars` | integer | 去掉 Markdown 标记后的总结正文长度 |
| `placeholder_count` | integer | “待核对/暂无/需补充”等占位词数量 |
| `provider_errors_json` | json/jsonb | 每个模型失败原因，注意不能包含 API Key |

同步更新：

- `server/lib/store.js` 的 JSON 迁移。
- `server/schema/supabase.sql` 的表结构。
- `serializeRecord()` 要把这些字段带给前端。

兼容旧数据：

- 旧 summary 没有 `quality_status` 时：
  - `model_provider === local-template` 且 `model_error` 非空，推断为 `fallback_template`。
  - `summary_markdown` 为空，推断为 `invalid`。
  - 其他推断为 `ai_ok`，但前端可显示“旧数据未评分”。

### 3.3 后端质量判定

新增函数建议放在 `server/lib/summary-quality.js`：

```js
function assessSummaryQuality(summary, transcriptText) {
  const cleanTranscript = stripWhitespace(transcriptText);
  const cleanSummary = stripMarkdown(summary.summaryMarkdown || '');
  const placeholderCount = countPlaceholders(cleanSummary);
  const modelProvider = summary.modelProvider || '';
  const hasProviderError = Boolean(summary.modelError);

  if (!cleanSummary || cleanSummary.length < 15) {
    return {
      qualityStatus: 'invalid',
      qualityReason: '总结正文为空或过短',
    };
  }

  if (modelProvider === 'local-template' && hasProviderError) {
    return {
      qualityStatus: 'fallback_template',
      qualityReason: '真实总结模型全部失败，系统仅生成临时模板',
    };
  }

  if (cleanTranscript.length >= 300 && cleanSummary.length < 120) {
    return {
      qualityStatus: 'low_information',
      qualityReason: '逐字稿较长，但总结正文过短',
    };
  }

  if (placeholderCount >= 3 && cleanSummary.length < 240) {
    return {
      qualityStatus: 'low_information',
      qualityReason: '总结主要由待核对占位内容组成',
    };
  }

  return {
    qualityStatus: 'ai_ok',
    qualityReason: '',
  };
}
```

注意：

- 不要只用字数判断。真实短录音可能只有一句话，应该允许生成短总结。
- `modelProvider === local-template && modelError` 是最明确的失败信号，应优先判定为 `fallback_template`。
- 质量判定只能防止“伪完成”，不能替代人工理解。

### 3.4 后端状态调整

修改 `server/app.js` 的 `summarizeExistingRecord()`：

当前行为：

```js
const summary = await generateSummary(...)
upsert summary
store.update(record, { status: 'completed' })
addProcessingEvent(..., 'completed', '总结已生成。')
```

建议行为：

1. 先生成 summary。
2. 调用 `assessSummaryQuality(summary, transcriptText)`。
3. 写入 summary + quality 字段。
4. 按质量决定 record 状态：

| 质量状态 | record.status | error_message | 前端含义 |
| --- | --- | --- | --- |
| `ai_ok` | `completed` | 空 | 转写和总结都完成 |
| `fallback_template` | `transcribed` | `AI 总结模型暂不可用，已保留逐字稿，可重试生成总结。` | 转写完成，总结失败 |
| `low_information` | `completed` 或 `transcribed` | 建议：如果模型真实成功则 `completed` 但展示黄色警告；如果本地模板则 `transcribed` | 总结质量不足 |
| `invalid` | `transcribed` | `总结生成结果为空，请重试。` | 转写完成，总结未完成 |

推荐第一期规则：

- 只要 `fallback_template` 或 `invalid`，就不要设为 `completed`。
- 设为 `transcribed`，表示“逐字稿成功，但总结未达标”。
- `low_information` 如果是真实模型返回，可以保留 `completed`，但前端显示质量提示。

事件文案：

- `ai_ok`：`总结已生成。`
- `fallback_template`：`AI 总结模型失败，已保留逐字稿，可稍后重试。`
- `invalid`：`总结结果为空，已保留逐字稿。`
- `low_information`：`总结已生成，但内容较少，建议人工核对。`

### 3.5 前端显示调整

修改 `sidepanel.js`：

#### 3.5.1 `statusLabel()`

已有 `transcribed: '已转写'`，但实际几乎没用。需要在 UI 中真正使用。

建议补充文案：

```js
transcribed: '已转写，待生成总结'
summary_failed: '总结失败'
```

第一期可以不新增 `summary_failed`，只用 `transcribed`。

#### 3.5.2 `hasSummaryContent()`

当前逻辑过宽：

```js
return Boolean(summary.summary_markdown || summary.overview_card_json || mindMapState(...).mindMap);
```

建议改成：

```js
function hasUsableSummaryContent(record) {
  const summary = record.summary || {};
  if (['fallback_template', 'invalid'].includes(summary.quality_status)) return false;
  return Boolean(summary.summary_markdown || hasOverviewCard(summary) || hasMindMap(summary));
}
```

用途：

- 下载总结按钮用 `hasUsableSummaryContent()`。
- “复制总结”按钮用 `hasUsableSummaryContent()`。
- 文件入口“总结”用 `hasUsableSummaryContent()`。

同时保留一个 `hasAnyGeneratedSummary()`，用于详情页显示“临时模板/低质量结果”。

#### 3.5.3 总结面板提示

在 `renderSummaryPanel()` 或 `renderMeetingResultPanel()` 顶部加质量提示：

| quality_status | 提示 |
| --- | --- |
| `fallback_template` | `AI 总结模型暂不可用，下面只是临时模板，不建议直接使用。请稍后重试或联系管理员检查模型池。` |
| `invalid` | `总结结果为空，请点击重新生成。逐字稿已保留。` |
| `low_information` | `本次总结信息较少，建议结合逐字稿核对。` |

按钮：

- `重新生成总结`
- `查看逐字稿`
- 管理员额外显示：`查看模型错误`

不要把 `model_error` 全量直接给普通员工看，避免出现内部服务 URL 或供应商细节。普通员工只看简短原因；管理员可展开详细错误。

#### 3.5.4 历史列表

历史列表每条记录增加总结质量小标：

- `AI总结完成`
- `总结待重试`
- `低信息总结`
- `临时模板`

如果 `record.status === 'transcribed'`，卡片主按钮应是“生成总结”，不是“打开完成结果”。

### 3.6 导出调整

修改 `server/lib/exporters.js`：

第一期推荐做法：

- `target === 'summary'` 且 `quality_status` 是 `fallback_template` / `invalid` 时，返回 400：`总结未生成可用内容，请先重新生成。`
- `target === 'all_files'` 可以包含逐字稿，但 summary 文件中写入一段明显说明：
  - `本记录 AI 总结未成功，仅包含临时模板或待核对信息。`

这样能避免同事把空壳 DOCX 当正式纪要发出去。

### 3.7 模型调用诊断和重试

当前模型池已经能记录最近测试和调用状态，但单条记录里只有 `model_error` 字符串。建议：

1. `generateSummary()` 返回结构化 provider 错误：

```js
{
  providerErrors: [
    { provider: 'easyai', model: 'gpt-5.5', message: '模型请求超时' },
    { provider: 'sub2api', model: 'gpt-5.5', message: '模型请求超时' },
    { provider: 'kimi', model: 'kimi-k2.6', message: '配额用尽' }
  ]
}
```

2. `summarizeExistingRecord()` 写入 `provider_errors_json`。
3. `record_processing_events` 在每个模型失败时写入一条事件：
   - `EasyAI GPT-5.5 失败，正在尝试下一个模型。`
   - `Kimi K2.6 失败，已降级为转写完成待重试。`

可选但推荐：

- 在 `POST /api/records/:id/summarize` 请求体支持 `providerId`，管理员可以指定某个模型重试。
- 默认重试仍按模型池优先级。

### 3.8 本地模板改进

本地模板应该只作为“故障兜底”，不要承担正式总结质量。但可以避免太空：

当前 `extractHighlights()` 只取前 6 个长句，`extractByKeywords()` 再按关键词过滤。如果 ASR 文本口语化、标点碎，容易只提到一句“不能贷了，两三天”。

改进建议：

1. `extractHighlights()` 不只取前 6 句，先取前 30 句，再按长度和重复度筛选。
2. `meeting_minutes` 模板的“关键结论”关键词增加：
   - `不能`、`不行`、`贵`、`不适合`、`配不上`、`贷不到`、`定位`
3. 如果页面标题或来源包含“红娘”，且用户没有显式选择会议纪要，默认模板建议切到 `matchmaker_profile` 或提示“是否按红娘客户画像生成”。

这属于 P1，不是 P0。P0 先解决“失败不伪装成功”。

## 4. 优化二：R2 上传 403 SignatureDoesNotMatch

### 4.1 新增问题现象

办公室同事上传录音后，页面最近处理进度显示：

```text
录音已上传，后台准备处理。
开始处理录音文件。
正在上传录音到 R2 存储。
R2 上传失败：HTTP 403
<Code>SignatureDoesNotMatch</Code>
<Message>The request signature we calculated does not match the signature you provided...</Message>
```

对应本地记录：

- 记录 ID：`a6372f88-a3b6-4acf-a268-5c5331c821f9`
- 员工：`泡泡`
- 文件名：`2026-06-18 09_18_40(1).mp3`
- 文件大小：`6438032`
- 状态：`failed`
- 失败阶段：上传到 Cloudflare R2
- R2 key：
  - `audio/2026-06-18/a6372f88-a3b6-4acf-a268-5c5331c821f9/2026-06-18 09_18_40(1).mp3`

这类失败发生在 DashScope 转写之前。录音已经上传到本机后端临时目录，但没有成功上传到 R2，因此后续不会生成公网临时音频链接，也不会进入转写。

### 4.2 初步根因判断

这次更像代码签名/对象路径问题，不像密钥整体失效。

证据：

1. 同一天更早的多条记录 R2 上传成功，例如：
   - `29cf42a6-6c7a-417f-a746-577162b8c666`
   - `ba6b0937-0c99-418b-b421-9fc9cc18848a`
   - `15faa8b5-471d-4941-be35-5c24daf58a96`
2. 失败记录的 R2 key 保留了原始文件名，里面包含空格和括号：`2026-06-18 09_18_40(1).mp3`。
3. 当前 `server/lib/r2.js` 的 `encodeObjectKey()` 使用 `encodeURIComponent()`，但 JavaScript 的 `encodeURIComponent()` 不会编码括号：

```text
encodeURIComponent('2026-06-18 09_18_40(1).mp3')
=> 2026-06-18%2009_18_40(1).mp3
```

4. AWS S3 Signature V4 / R2 兼容签名的 Canonical URI 应使用更严格的 URI 编码：除 `A-Z a-z 0-9 - . _ ~` 外，其他字符都应百分号编码。`(` 和 `)` 应为 `%28`、`%29`。
5. 当前测试只验证 URL 大致存在签名，没有覆盖括号、感叹号、星号、单引号等特殊字符。

因此第一嫌疑：

- R2 预签名 URL 中 object key 的 Canonical URI 编码不符合 S3 SigV4 规范；
- 或者最终请求路径和签名计算路径在特殊字符上不一致，导致 R2 返回 `SignatureDoesNotMatch`。

第二个需要顺手修的风险：

- R2 对象 key 直接使用原始文件名，可能把手机号、客户名、微信导出的私密标题写入对象路径；即使对象私有，这也不是好习惯。

### 4.3 要改的文件

优先修改：

- `server/lib/r2.js`
- `server/lib/adapters.test.js`
- `server/app.js`
- `录音助手上线反馈优化方案-2026-06-18.md`

可选同步：

- `README.md`
- `server/schema/supabase.sql` 不一定需要改字段，但如果新增错误分类字段，要同步。

### 4.4 修复方案 A：严格 SigV4 URI 编码

在 `server/lib/r2.js` 增加专用编码函数：

```js
function awsUriEncode(value) {
  return encodeURIComponent(String(value))
    .replace(/[!'()*]/g, (char) =>
      `%${char.charCodeAt(0).toString(16).toUpperCase()}`
    );
}
```

替换当前：

```js
function encodePathSegment(value) {
  return encodeURIComponent(value).replace(/%2F/g, '/');
}

function encodeObjectKey(key) {
  return String(key)
    .split('/')
    .map((segment) => encodeURIComponent(segment))
    .join('/');
}
```

建议改成：

```js
function encodePathSegment(value) {
  return awsUriEncode(value).replace(/%2F/g, '/');
}

function encodeObjectKey(key) {
  return String(key)
    .split('/')
    .map((segment) => awsUriEncode(segment))
    .join('/');
}
```

注意：

- 不要用 `encodeURI()`。
- 不要把 `/` 编成 `%2F`，因为 object key 中的 `/` 是路径分隔。
- query string 也建议使用同一套严格编码函数，至少要补测试防止未来踩坑。

### 4.5 修复方案 B：R2 key 不再使用原始文件名

在 `server/app.js` 的 `processUploadedRecord()` 中，当前 R2 key 是：

```js
storageKey = storageKey || `audio/${date}/${record.id}/${record.original_file_name || `${record.id}.mp3`}`;
```

建议改成：

```js
const ext = safeExtension(record.original_file_name || '') || 'mp3';
storageKey = storageKey || `audio/${date}/${record.id}/source.${ext}`;
```

或者：

```js
storageKey = storageKey || `audio/${date}/${record.id}/${record.id}.${ext}`;
```

推荐第一种：`source.<ext>`。

理由：

- 避免特殊字符导致签名问题。
- 避免手机号、客户名、导出文件名进入 R2 对象路径。
- 原始文件名仍保存在 `audio_records.original_file_name`，前端展示不受影响。

迁移兼容：

- 老记录已有 `r2_key` 的，不要改。
- 只有新上传记录使用新 key。

### 4.6 修复方案 C：错误提示更友好

当前前端直接显示 R2 XML 错误，普通同事看不懂。

建议后端新增错误分类函数：

```js
function classifyR2UploadError(error) {
  const message = String(error?.message || error || '');
  if (message.includes('SignatureDoesNotMatch')) {
    return {
      publicMessage: '录音上传到云存储失败，可能是文件名特殊字符或存储签名配置问题。录音已到达后端，请联系管理员重试。',
      technicalCode: 'r2_signature_mismatch',
    };
  }
  if (message.includes('AccessDenied') || message.includes('InvalidAccessKeyId')) {
    return {
      publicMessage: '云存储密钥或权限异常，请联系管理员检查 R2 配置。',
      technicalCode: 'r2_permission_error',
    };
  }
  return {
    publicMessage: '录音上传到云存储失败，请稍后重试或联系管理员。',
    technicalCode: 'r2_upload_failed',
  };
}
```

落地方式：

- `record.error_message` 存用户可读文案。
- `record_processing_events.metadata_json` 存技术错误摘要和 `technicalCode`。
- 管理员点击“复制处理诊断”时仍能拿到原始错误片段。
- 普通员工页面不直接显示 XML。

### 4.7 失败后重试策略

当前失败后，页面提供“重新转写”。但这条记录的音频已经在本机 `uploads` 目录，理论上无需同事重新上传。

建议：

1. 如果记录 `status=failed` 且本地音频文件存在：
   - “重新转写”按钮调用现有 `/api/records/:id/transcribe`。
   - 后端重新进入 `enqueueRecordProcessing(record)`。
2. 修复 R2 key/签名后，管理员或同事点“重新转写”即可继续。
3. 如果本地文件已被清理，才提示“请重新上传录音文件”。

需要确认：

- `processUploadedRecord()` 使用本地路径：`uploads/<record.id>.<ext>`。
- 失败记录本地文件未删除。
- `cleanupMultipartUpload(upload)` 只清理临时 `.part`，不会删最终移动后的文件。

## 5. 优化三：局域网插件更新机制

### 5.1 现状

当前录音助手是 Chrome 扩展 + 局域网后端：

- `manifest.json` 的版本是 `0.1.0`。
- 没有 `update_url`。
- 扩展是开发模式“加载已解压的扩展程序”方式使用时，浏览器不会像 Electron 应用那样自动下载 zip、替换目录、重启。
- 后端配置变更，例如 DashScope/R2/模型池，已经可以通过后端统一生效；这类不需要更新插件。
- 前端扩展代码变更，例如 `sidepanel.js`、`background.js`、`manifest.json` 权限变更，需要同事浏览器端更新扩展。

### 5.2 ztools 的机制对照

`/Users/lixin/AI code/ztools` 的局域网更新机制是：

- 打包产物：
  - `dist-lan/latest.yml`
  - `dist-lan/update-darwin-universal-<版本>.zip`
  - `dist-lan/update-win32-x64-<版本>.zip`
- 更新源：
  - `resources/update-config.json` 里配置 `http://lixindemac-studio.local:8083`
- 应用启动后检查 `latest.yml`。
- 下载对应平台的 zip。
- 启动独立 updater 替换 `app.asar`，再重启应用。

Chrome 扩展没有这个独立 updater。扩展 JavaScript 也不能自己覆盖安装目录。因此不能 1:1 复刻 ztools 的“自动替换并重启”。

### 5.3 推荐分三档实现

#### A 档：短期可落地 - 局域网版本提示 + 下载新版包

适合当前“加载已解压扩展”的办公室使用方式。

新增发布目录：

```text
releases/extension/
  latest.json
  voice-to-word-extension-0.1.1.zip
  voice-to-word-extension-0.1.1.sha256
```

`latest.json` 示例：

```json
{
  "name": "大宜宾录音助手",
  "version": "0.1.1",
  "minSupportedVersion": "0.1.0",
  "releasedAt": "2026-06-18T00:00:00.000Z",
  "changelog": [
    "修复 AI 总结失败但显示完成的问题",
    "修复特殊文件名导致 R2 上传失败的问题",
    "管理员历史记录支持按人展开"
  ],
  "downloadUrl": "http://lixindemac-studio.local:8127/releases/extension/voice-to-word-extension-0.1.1.zip",
  "sha256": "..."
}
```

后端新增接口：

```text
GET /api/extension/latest
GET /releases/extension/latest.json
GET /releases/extension/voice-to-word-extension-<version>.zip
```

前端逻辑：

1. 侧边栏启动时读取：
   - 本地版本：`chrome.runtime.getManifest().version`
   - 后端最新版本：`GET /api/extension/latest`
2. 如果 `latest.version > currentVersion`：
   - 顶部显示“发现新版插件”
   - 展示更新内容
   - 提供“下载新版插件包”
   - 提供“打开扩展管理页/查看更新步骤”
3. 如果 `minSupportedVersion > currentVersion`：
   - 阻止上传/监听等高风险动作
   - 只允许查看历史和下载旧数据
   - 明确提示“当前插件版本过旧，请更新后继续使用”

优点：

- 改动小。
- 不影响现有同事使用。
- 不需要企业策略。

缺点：

- 不能静默安装。
- 同事仍需要按提示替换本地扩展目录，并在 `chrome://extensions` 点重新加载。

#### B 档：中期推荐 - 打包 CRX + update_url

适合想接近自动更新，但仍在局域网内自托管的场景。

Chrome 扩展官方机制支持：

- `manifest.json` 增加 `update_url`
- 服务器返回更新 manifest XML
- 新版本 `.crx` 必须使用同一私钥签名
- 浏览器每隔一段时间检查更新
- 也可以用 `chrome.runtime.requestUpdateCheck()` 做主动检查，但频繁调用会被浏览器限制；更新下载后仍需要扩展空闲或 `chrome.runtime.reload()` 才能应用

发布目录：

```text
releases/extension-crx/
  updates.xml
  voice-to-word-extension-0.1.1.crx
```

`manifest.json`：

```json
{
  "version": "0.1.1",
  "update_url": "http://lixindemac-studio.local:8127/releases/extension-crx/updates.xml"
}
```

`updates.xml` 示例：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<gupdate xmlns="http://www.google.com/update2/response" protocol="2.0">
  <app appid="固定扩展ID">
    <updatecheck
      codebase="http://lixindemac-studio.local:8127/releases/extension-crx/voice-to-word-extension-0.1.1.crx"
      version="0.1.1" />
  </app>
</gupdate>
```

关键要求：

1. 首次安装也要使用同一 CRX 或同一私钥打包，否则扩展 ID 会变。
2. `appid` 必须匹配实际扩展 ID。
3. `version` 必须严格递增，不能回退。
4. 每次发布 `.crx` 必须用同一私钥。
5. 需要先在测试机验证内网 HTTP 是否被当前 Chrome 接受；如受限，则改用内网 HTTPS 或企业策略。

优点：

- 最接近 Chrome 原生自动更新。
- 不需要每次手动替换目录。

缺点：

- 初次安装流程要重做。
- 私钥必须妥善备份，丢失会导致无法原地更新。
- 自托管 CRX 在不同 Chrome 策略环境下可能有安装限制，需要实机验证。

#### C 档：长期最稳 - Chrome 企业策略/统一管理

适合公司内部长期推广。

方案：

- 用 Chrome Enterprise/企业策略把扩展 ID 和 update URL 配置为强制安装或推荐安装。
- 员工不需要手动加载解压扩展。
- 管理员统一控制版本和更新。

优点：

- 最适合办公室规模化管理。
- 自动更新能力最完整。

缺点：

- 需要公司设备管理或策略下发能力。
- 首次配置成本最高。

### 8.1 总结质量闸门风险

风险：某些真实短录音会被误判为低质量。

控制：

- 不要对短逐字稿严格卡字数。
- `fallback_template + model_error` 才是 P0 的强判定。
- `low_information` 先只做提示，不阻断。

### 8.2 状态变更风险

风险：新增/启用 `transcribed` 状态后，前端进度条和历史筛选漏处理。

控制：

- `statusLabel()`、`progressStepsFor()`、`isInProgress()`、历史 badge 都要一起检查。
- `transcribed` 不算进行中，但详情页要显示“可重试生成总结”。

### 8.3 R2 上传修复风险

风险：修改 R2 key 后，新旧记录对象路径不同。

控制：

- 已有 `r2_key` 的老记录不迁移、不改名。
- 只影响新上传记录。
- 原始文件名继续保存在 `original_file_name`，展示和搜索不受影响。
- 先不要轮换 R2 Key/Secret；当前证据更像特殊文件名签名问题。

风险：严格编码改动影响中文文件名。

控制：

- 测试同时覆盖中文、空格、括号、感叹号、单引号。
- 手动上传中文名和括号名各一次。

### 8.4 更新机制风险

风险：同事以为能像 ztools 一样完全自动更新，但解压扩展做不到。

控制：

- 第一阶段文案明确：“发现新版，需要按步骤更新扩展”。
- 只有 CRX/企业策略路线才承诺自动更新。
- 每次发布都保留旧 zip，方便回滚。

### 8.5 分组历史风险

风险：管理员以为折叠后记录消失。

控制：

- 顶部显示总数。
- 搜索时显示“命中 X 人 / Y 条记录”。
- 有进行中/失败/总结待重试的组默认展开。
