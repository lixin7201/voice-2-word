# 大宜宾录音助手可用性修复与界面收敛需求文档

更新时间：2026-06-17

本文只做问题定位、需求收敛和验收标准，不执行功能代码修改。等确认本文口径后，再进入实现阶段。

## 1. 本次目标

当前插件的核心问题不是单个按钮样式，而是“输入、长任务、状态展示、导出”几条主链路没有形成稳定可用闭环。本次优化目标是：

1. 个人资料页输入框必须可正常输入、保存、再次打开仍保留。
2. 识别到网页录音后，点击生成会议纪要不能再被前端 15 秒超时误杀。
3. 处理状态压缩为一个清晰状态：已上传 -> 转写中 -> 总结中 -> 完成。
4. 每完成一步，下方对应文件立即可点开看：录音、逐字稿、总结。
5. 下载区只保留两个入口：下载逐字稿、下载总结；文件格式全部以下拉框展示。
6. 下载总结选择 PDF/DOCX 时，默认把总结卡片、文字总结、思维导图保存为同一份文件。
7. 会议纪要和跟单信息两种生成类型使用不同结果界面，但都保留带时间戳的逐字稿，点击时间戳能跳转播放。
8. 不再把结果页做成复杂“生成文件区”或多层工作台，界面参考用户提供的图 2：上传区、生成设置、结果区三块清楚即可。

## 3. 当前核心问题结论

### P0-1：个人资料输入框不能输入

用户现象：

- 点击头像进入资料页后，任何输入框都无法输入内容。

当前代码证据：

- `sidepanel.js:918-980` 已经渲染了个人资料、头像、密码表单，输入框本身没有 `disabled`。
- `server/app.js:167-193` 已有个人资料、头像、改密码接口。
- `sidepanel.js:69-73` 会监听 Chrome 后台 `STATE_UPDATE`。
- `sidepanel.js:1721-1733` 当后台识别到录音或报错时，会触发 `scheduleRender()`。
- `sidepanel.js:96-110` 的 `render()` 会重写整个 `app.innerHTML`。

判断：

- 输入框不是因为 HTML 禁用导致不能输入。
- 更可疑的原因是：Chrome 后台在持续推送录音状态，前端收到 `STATE_UPDATE` 后整页重渲染；用户刚输入的字符立刻被旧的 `appState.profile` 覆盖，所以体感就是“输入不了”。
- 这个问题必须在真实 Chrome 扩展侧栏里复现和验证，单纯看静态 HTML 不够。

需求：

1. 资料页、设置页、员工页、详情编辑页有输入焦点时，后台录音状态更新不得触发整页重渲染。
2. 个人资料页必须维护编辑草稿，输入时先写入草稿，保存成功后再刷新 `currentUser/profile`。
3. 后台发现录音时，如果当前不在首页、历史、上传、监听页面，只更新轻量提示，不打断当前表单。
4. 保存按钮点击后才允许进入提交状态，提交期间可显示保存中，但不能清空用户已输入内容。

验收：

1. 点击头像进入个人资料。
2. 花名、一句话自我介绍、AI 生成偏好、旧密码、新密码、确认密码都能连续输入中文、英文、数字。
3. 在资料页输入过程中，让插件后台识别到网页录音，输入框内容不能被清空，光标不能丢失。
4. 保存个人资料后刷新侧栏，再次进入资料页仍能看到保存内容。
5. 改密码表单能输入，错误密码给明确提示，正确密码保存成功。

### P0-2：点击生成会议纪要直接请求超时

用户现象：

- 已识别到录音，点击生成会议纪要后提示“请求超时，请确认后端服务是否正常。”

当前代码证据：

- `sidepanel.js:5` 默认请求超时是 `15000` 毫秒。
- `sidepanel.js:2030-2043` 的 `summarizeRecord()` 调用 `/api/records/:id/summarize` 时没有传长任务超时时间。
- `sidepanel.js:2510-2534` 的 `api()` 对所有请求套用了默认 15 秒超时，超时后直接抛“请求超时”。
- `server/app.js:354-368` 的 `/api/records/:id/summarize` 会 `await summarizeExistingRecord(...)`，也就是服务端同步等 LLM 总结完成后才响应。

判断：

- 这是明确的长任务模型错误：生成会议纪要本来可能超过 15 秒，但前端把它当普通接口处理。
- ai-dayibin 的 `AgentRecordingSummaryWorkspace.tsx:466-487` 生成纪要没有套一个全局 15 秒 abort。
- ai-dayibin 的 `AgentRecordingSummaryWorkspace.tsx:440-458` 对录音处理有“启动任务 -> 轮询附件状态 -> 完成后刷新详情”的模式。

需求：

优先方案：

1. `/api/records/:id/summarize` 改为启动后台总结任务后尽快返回。
2. 返回时记录状态为 `summarizing`，前端进入“总结中”状态。
3. 前端通过现有 `/api/records/:id` 轮询状态，不阻塞在一个长请求上。
4. 总结完成后状态变为 `completed`，总结文件入口自动可点击。
5. 失败时状态变为 `failed`，显示失败原因和“重新生成”按钮。

临时兜底方案：

- 如果暂时不改后台任务模型，至少给 `/summarize` 单独设置 10 分钟级别超时。但这只是止血，不是最终方案，因为用户仍会看到按钮卡住。

验收：

1. 已有逐字稿的录音点击“生成会议纪要”后，页面立刻进入“总结中”，不出现 15 秒超时。
2. 生成过程中可以离开详情页再回来，状态仍准确。
3. 生成完成后，会议纪要内容、总结卡片、思维导图都出现。
4. LLM 慢、网络慢、失败时，都能在状态区看到明确结果，而不是假死或误报后端异常。

### P0-3：当前状态面板太复杂，且不是用户需要的主界面

用户现象：

- 截图中的“最近处理进度”事件列表很冗余。
- 用户明确要求压缩成一个状态显示：已上传、转写中、总结中。

当前代码证据：

- `sidepanel.js:2345-2368` 会渲染 `progress-events` 和“最近处理进度”。
- `server/sidepanel-ui.test.js:538` 目前还在测试“record progress renders recent processing events”，这会把冗余 UI 固化下来。

需求：

1. 主界面只显示一个紧凑状态条：
   - 已上传
   - 转写中
   - 总结中
   - 完成
   - 失败
2. 状态条下最多显示一句当前说明，例如“正在生成会议纪要，请稍等”。
3. 不再展示多条事件流水。
4. 事件流水可以留在数据里，用于调试或后台日志，但不在普通用户主界面展示。
5. 完成一个阶段后，下方文件入口立即解锁：
   - 已上传：录音可播放或打开。
   - 转写完成：逐字稿可打开，时间戳可跳转。
   - 总结完成：总结可打开，下载总结可用。

验收：

1. 截图中那种五条事件列表不再出现在主界面。
2. 用户只需要看一眼就知道当前处于哪个阶段。
3. 出错时显示红色错误状态和重试动作。
4. 测试要从“渲染最近处理进度”改成“只渲染紧凑状态，不渲染事件流水”。

## 4. 参考图 2 的界面收敛要求

### 4.1 页面结构

保留三块主结构：

1. 上传/识别区
   - 上传录音文件，开始转写。
   - 网页识别到的当前录音显示为一行。
   - 当前录音行只需要：播放/文件名/大小/时长/状态/删除。

2. 转写设置区
   - 只保留两个大选项：
     - 会议纪要
     - 跟单信息
   - 默认设置只保留“设为默认”和“保存设置并应用”。
   - 其它模板复杂度不要露出给普通用户。

3. 转写结果区
   - 顶部保留结果类型、字数、预计阅读、生成时间。
   - 右侧只保留复制、下载逐字稿、下载总结、分享等必要动作。
   - 结果区内部根据“会议纪要/跟单信息”切换内容结构。

### 4.2 删除或隐藏的内容

1. 不再单独展示“下载总结卡片”。
2. 不再单独展示“下载思维导图”。
3. 不再展示“下载全部文件 ZIP”。
4. 不再出现复杂的“生成的文件”区域。
5. 不再出现多层 Tab：顶部导航、详情页 Tab、工作台 Tab 不能叠在一起。
6. 不再把处理事件流水作为主内容展示。

### 4.3 下载入口

只保留两个下载入口：

1. 下载逐字稿
   - 格式下拉：Markdown、TXT、DOCX、PDF。
   - 内容只包含逐字稿。
   - 逐字稿必须保留时间戳。

2. 下载总结
   - 格式下拉：Markdown、TXT、DOCX、PDF。
   - Markdown：包含总结卡片文本、文字总结、思维导图文本。
   - TXT：纯文本版本。
   - DOCX/PDF：必须包含视觉化总结卡片、文字总结、视觉化思维导图，并保存在同一份文件。

### 4.4 当前导出与 ai-dayibin 差距

当前项目：

- `server/lib/exporters.js:35-45` 总结导出文本里已经包含总结卡片、文字重点总结、思维导图。
- `server/lib/exporters.js:311-340` DOCX/PDF 仍是简化文字生成，没有把总结卡片和思维导图作为图片/视觉模块嵌入。

ai-dayibin：

- `local-agent-attachments-handler.ts:1463-1499` 会先生成总结卡片图片和思维导图图片。
- PDF 走 `buildSummaryExportHtml(...)` + `writeHtmlPdf(...)`。
- DOCX 走 `writeSummaryDocx(...)`，并传入 `cardImagePath`、`mindMapImagePath`。

需求：

1. 当前项目的“下载总结 DOCX/PDF”要对齐 ai-dayibin：
   - 生成总结卡片视觉图。
   - 生成思维导图视觉图。
   - DOCX/PDF 中顺序为：标题 -> 总结卡片 -> 文字总结 -> 思维导图。
2. 用户不需要手动选择“带卡片”或“带思维导图”，选择 DOCX/PDF 即默认包含。
3. Markdown/TXT 可以保留文本化版本，不强制嵌入图片。

验收：

1. 下载总结 PDF 打开后，第一页能看到总结卡片视觉模块。
2. 同一份 PDF 内能看到文字总结和思维导图。
3. 下载总结 DOCX 打开后，也包含总结卡片图片、文字总结、思维导图图片。
4. 下载逐字稿不会混入总结内容。

### 5.2 会议纪要界面

会议纪要结果区应展示：

1. 会议概览。
2. 关键要点。
3. 待办事项。
4. 风险/待确认事项。

展示方式：

1. 客户/对象名称。
2. 业务类型。

4. 需求或意向。
5. 关键信息。
6. 下一步动作。
7. 负责人/时间。
8. 风险或待确认。

展示方式：

- 如果 AI 无法判断某个字段，显示“待核对”，不要编造。
- 跟单信息仍然要能下载总结，下载内容包含字段化跟单总结、文字说明、思维导图或关系图。

## 6. 生成流程需求

### 6.1 网页识别录音

流程：

1. 插件从当前网页识别候选录音。
2. 只把最可能的当前录音放在主位置。
3. 其它候选折叠，默认不干扰用户。
4. 用户点击“生成会议纪要”或“生成跟单信息”。
5. 系统上传音频并创建记录。
6. 页面进入状态条：
   - 已上传
   - 转写中
   - 总结中
   - 完成

验收：

1. 用户从 Plaud 页面进入插件，能看到当前录音。
2. 点击一次即可进入处理状态。
3. 不出现“请求超时”误报。
4. 上传失败时明确提示网页登录态、地址过期、跨域读取失败或手动上传兜底。

### 6.2 手动上传录音

流程：

1. 用户选择本地 mp3/wav/m4a。
2. 选择会议纪要或跟单信息。
3. 点击上传并生成。
4. 进入同一套状态条。
5. 每一步完成后文件入口解锁。

验收：

1. 200MB 内文件可上传。
2. 上传中有明确状态。
3. 上传后页面不会停留在“暂无逐字稿待补充”不刷新。
4. 失败时可以重新上传或重新转写。

### 6.3 重新生成总结

流程：

1. 已有逐字稿时，用户可以切换会议纪要/跟单信息。
2. 点击重新生成。
3. 记录状态进入“总结中”。
4. 轮询完成后切换到对应结果界面。

验收：

1. 切换类型后生成的界面结构不同。
2. 会议纪要不会显示跟单字段。
3. 跟单信息不会显示会议纪要模板化段落。
4. 逐字稿保留不变。

## 7. 技术实现要求

### 7.1 长任务处理

必须避免普通接口长时间挂起。

建议最小改法：

1. 保留现有 `audio_records.status` 作为任务状态源。
2. `/api/records/:id/summarize`：
   - 更新记录为 `summarizing`。
   - 追加一条内部处理事件。
   - 启动 `summarizeExistingRecord(...)` 后台 Promise。
   - 立即返回当前记录。
3. 后台 Promise 成功：
   - 写入 summary、followup、AI 标题。
   - 状态改为 `completed`。
4. 后台 Promise 失败：
   - 状态改为 `failed`。
   - 写入 `error_message`。
5. 前端调用 summarize 后不等待总结完成，只打开详情并开始轮询。

不建议：

- 不建议只把全局超时从 15 秒改成很大，因为其它普通接口会变得难以失败反馈。
- 不建议新增复杂队列系统，除非现有 Promise 后台任务无法满足。

### 7.2 渲染与输入稳定

必须避免“后台状态更新导致表单重渲染”。

建议最小改法：

1. `handleBackgroundState` 只在首页、历史、上传、监听页触发页面级重渲染。
2. 当前在 `profile/settings/employees/detail` 且有输入焦点时，不执行整页 `render()`。
3. 个人资料表单使用 `appState.profileDraft`：
   - 进入资料页时初始化草稿。
   - input/textarea/change 时更新草稿。
   - 保存成功后清理草稿并刷新 profile。
4. 其它编辑态也要遵守同一规则：
   - 标题编辑。
   - 跟单备注编辑。
   - 设置密钥编辑。
   - 员工创建表单。

### 7.3 状态与文件入口

主状态从记录字段派生：

1. `uploaded` -> 已上传。
2. `transcribing` -> 转写中。
3. `summarizing` -> 总结中。
4. `completed` -> 完成。
5. `failed` -> 失败。

文件入口启用规则：

1. 音频入口：`audioUrl` 存在即可启用。
2. 逐字稿入口：`transcript.corrected_text/raw_text/segments_json` 任一存在即可启用。
3. 总结入口：`summary.summary_markdown` 或 `summary.overview_card_json` 存在即可启用。
4. 下载总结：总结入口启用后才启用。
5. 下载逐字稿：逐字稿入口启用后才启用。

### 7.4 下载实现

前端：

1. 保留“下载逐字稿”下拉。
2. 保留“下载总结”下拉。
3. 下拉内展示 Markdown、TXT、DOCX、PDF。
4. 移除总结卡片、思维导图、全部文件 ZIP 的单独按钮。

后端：

1. `target=summary` 保持对外语义不变。
2. 当 `format=docx/pdf` 时，summary 导出必须包含视觉化总结卡片和视觉化思维导图。
3. 可参考 ai-dayibin：
   - 先把卡片/导图渲染成图片。
   - PDF 使用 HTML 渲染。
   - DOCX 嵌入图片。
4. `target=transcript` 只导出逐字稿。

## 9. 完成定义

只有以下全部满足，才可以告诉用户“插件现在可用”：

1. 资料页所有输入框真实可输入。
2. 生成会议纪要不再出现 15 秒请求超时。
3. 处理状态简洁清楚。
4. 每阶段完成后，对应文件可点开。
5. 会议纪要和跟单信息界面确实不同。
6. 逐字稿时间戳点击能跳转。
7. 下载入口只剩下载逐字稿和下载总结。
8. PDF/DOCX 总结包含总结卡片、文字总结、思维导图。
9. 自动化测试通过。
10. 真实 Chrome 扩展侧栏人工验收通过。

## 10. 建议实施顺序

1. 先修资料页输入被重渲染打断的问题。
2. 再修生成纪要长任务模型，去掉 15 秒误超时。
3. 再压缩状态 UI，移除事件流水主展示。
4. 再调整结果区和下载区。
5. 最后补齐 PDF/DOCX 视觉导出。
6. 每完成一阶段都跑测试，最后做 Chrome 扩展人工验收。

这样做的原因：

- 输入不可用和生成超时是 P0，不先修这两项，界面再漂亮也不可用。
- 状态 UI 依赖后端状态模型，先修长任务再改 UI 最稳。
- 导出视觉化涉及 PDF/DOCX，风险比纯前端 UI 高，适合放在最后集中验收。
