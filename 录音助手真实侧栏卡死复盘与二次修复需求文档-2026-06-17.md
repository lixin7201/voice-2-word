# 大宜宾录音助手真实侧栏卡死复盘与二次修复需求文档

更新时间：2026-06-17

本文基于用户在真实 Chrome 扩展侧栏中的手动测试反馈、两张截图，以及当前代码检查结果整理。本文只做问题复盘、根因判断、修复需求和验收标准，不直接代表已修复完成。

## 1. 用户反馈现象

### 1.1 个人资料页完全不可用

用户操作：

1. 打开个人资料页面。
2. 选择头像文件后点击上传。
3. 点击自我介绍、旧密码、新密码等输入框。

实际现象：

1. 头像选择后上传没有任何反应。
2. 自我介绍、旧密码、新密码等输入框都无法正常点击输入。
3. 用户体感是整个资料页卡死。

### 1.2 插件会消失，需要在扩展管理页重新加载

用户操作：

1. 在侧栏连续测试资料页、录音识别、生成纪要等功能。
2. 插件后续不见了。
3. 需要到 `chrome://extensions` 点击“重新加载”。

实际现象：

1. 侧栏状态不稳定。
2. 插件没有在界面里给出明确错误。
3. 用户只能通过重新加载扩展恢复。

### 1.3 识别到录音后点击生成纪要没有反应

用户操作：

1. 在 Plaud 网页识别到当前录音。
2. 点击“生成纪要”。
3. 再点击“手动上传”“首页”等按钮。

实际现象：

1. 点击“生成纪要”后没有状态提示。
2. 其它按钮也像失效一样没有反应。
3. 用户判断插件很容易卡死。

## 2. 已确认的关键根因

### P0 根因 1：`body[data-view]` 被误绑定成导航按钮，导致任何点击都触发整页重渲染

当前代码证据：

- `sidepanel.js:98-100`

```js
function render() {
  document.body.dataset.view = appState.view;
```

- `sidepanel.js:1544-1546`

```js
function bindCommon() {
  document.querySelectorAll('[data-view]').forEach((button) => {
    button.addEventListener('click', async () => {
```

问题解释：

1. `render()` 把当前视图写到 `document.body.dataset.view`。
2. `bindCommon()` 又用全局选择器 `[data-view]` 查找所有导航项。
3. 因为 `body` 也有 `data-view`，所以 `body` 被当成一个导航按钮绑定了点击事件。
4. 用户点击任何输入框、按钮、空白区域时，事件都会冒泡到 `body`。
5. `body` 的错误导航监听器会立即执行 `render()`，重写整个 `#app.innerHTML`。

这可以直接解释：

1. 输入框点不进去：刚聚焦就冒泡到 `body`，页面重渲染，焦点丢失。
2. 头像上传没反应：选择文件或点击上传后，页面被重渲染，文件选择和按钮状态可能被清掉。
3. 生成纪要没反应：点击按钮后，事件冒泡触发额外重渲染，用户看不到稳定的处理中状态。
4. 首页、手动上传等按钮后续没反应：页面处在频繁重渲染和异步任务交错状态，所有交互都不可信。

这是本次最优先修复项。

### P0 根因 2：所有操作共用 `runBusy()` 整页重渲染，容易打断表单和异步动作

当前代码证据：

- `sidepanel.js:2739-2748`

```js
async function runBusy(task, shouldRender = true) {
  appState.busy = true;
  if (shouldRender) render();
  try {
    await task();
  } catch (error) {
    setStatus(error.message || String(error), 'error');
  } finally {
    appState.busy = false;
    if (shouldRender) render();
  }
}
```

问题解释：

1. 每个动作开始前默认整页重渲染。
2. 对头像、个人资料、候选录音上传这类依赖当前 DOM 表单或文件输入的动作，整页重渲染风险很高。
3. 如果再叠加 P0 根因 1 的 `body` 误绑定，用户一次点击可能触发多次重渲染。
4. 异步任务过程中没有独立 action key，界面无法判断是哪一个按钮正在执行。

必须改成“局部动作状态”，不能所有按钮都靠整页重渲染表达忙碌。

### P0 根因 3：网页录音读取缺少候选卡片级状态、超时和失败恢复

当前代码证据：

- `sidepanel.js:1892-1928`
- `sidepanel.js:1930-1938`

```js
async function readCandidateBlob(candidate) {
  try {
    const response = await fetch(candidate.url, { credentials: 'include' });
```

问题解释：

1. 点击“生成纪要”后，侧栏直接 `fetch(candidate.url)` 读取 Plaud 录音。
2. Plaud 录音 URL 可能是临时地址、跨域地址、鉴权地址或需要网页上下文 Cookie。
3. 当前没有候选卡片内的“读取中 / 上传中 / 失败原因 / 改用手动上传”状态。
4. 错误只写到顶部状态栏，用户视线在按钮附近时会误以为没反应。
5. `fetch(candidate.url)` 没有单独短超时、取消按钮、重试按钮。

必须把“读取网页录音”视为一个高失败率步骤，而不是普通按钮点击。

### P0 根因 4：真实扩展环境缺少前端错误边界和诊断面板

当前问题：

1. 如果侧栏 JS 抛出未捕获异常，用户只看到插件消失或按钮失效。
2. 页面没有显示最近 JS 错误。
3. 没有把错误写入 `chrome.storage.local` 供重新打开后查看。
4. 没有“复制诊断信息”按钮。

这会导致开发者无法从用户反馈里快速确认是：

1. 侧栏渲染异常。
2. 背景 service worker 异常。
3. API 请求异常。
4. 网页录音读取异常。
5. Chrome 扩展上下文失效。

## 3. 二次修复目标

本轮不再优先做视觉细节，而是先把侧栏做成“不会因为一次失败变成全局失灵”的稳定工具。

目标：

1. 点击任意输入框不能触发整页重渲染。
2. 个人资料页输入、头像上传、密码输入必须在真实扩展侧栏可用。
3. 点击“生成纪要”必须立即在当前候选卡片内显示动作状态。
4. 网页录音读取失败必须在卡片内显示明确原因和手动上传入口。
5. 任何按钮失败都不能让其它导航按钮失效。
6. 插件侧栏发生 JS 错误时，界面必须显示可复制的错误诊断。
7. 自动化测试必须覆盖真实 DOM 连续点击，不只测 HTML 字符串。

## 4. 必改需求

### P0-1：修复 `body[data-view]` 误绑定导航

必须修改：

1. `sidepanel.js:99` 不再使用 `document.body.dataset.view`。
2. 改为 `document.body.dataset.currentView = appState.view`，或完全不用 body dataset。
3. `bindCommon()` 的导航选择器必须从：

```js
document.querySelectorAll('[data-view]')
```

改为：

```js
document.querySelectorAll('button[data-view]')
```

或者：

```js
document.querySelectorAll('[data-nav-view]')
```

4. 导航按钮点击处理必须接收 `event`，并执行：

```js
event.preventDefault();
event.stopPropagation();
```

5. 不允许 `body`、`main`、`section` 等容器元素拥有会被导航绑定器捕获的 `data-view`。

验收：

1. 在个人资料页点击输入框，不触发 `render()`。
2. 输入框能连续输入中文、英文、数字。
3. `document.querySelectorAll('body[data-view]').length` 必须为 0。
4. 自动化测试中点击 `#profile-bio` 后，`document.activeElement.id === 'profile-bio'`。
5. 点击输入框后 `#app.innerHTML` 不被重写。

### P0-2：重构通用按钮绑定，避免一个错误拖死全局

必须修改：

1. `bindCommon()` 只绑定全局导航和退出登录。
2. `bindCurrentView()` 中每个功能区的绑定必须使用更具体选择器，例如：
   - `button[data-action="upload-candidate"]`
   - `button[data-action="upload-avatar"]`
   - `button[data-action="save-profile"]`
3. 所有按钮处理函数必须接收 `event` 并 `preventDefault()`。
4. 点击按钮后，不允许因为事件冒泡触发第二个无关动作。
5. 增加 `safeBind()` 或 `safeHandler()` 包装，捕获每个按钮处理函数里的同步异常。

建议实现：

```js
function onClick(selector, handler) {
  document.querySelectorAll(selector).forEach((node) => {
    node.addEventListener('click', async (event) => {
      event.preventDefault();
      event.stopPropagation();
      try {
        await handler(event, node);
      } catch (error) {
        reportClientError(error, { selector });
      }
    });
  });
}
```

验收：

1. 一个按钮报错后，首页/上传/历史/配置仍可点击。
2. 控制台无未捕获 `TypeError`。
3. 点击输入框不会触发导航。
4. 点击“生成纪要”不会同时触发 body 或父级导航监听。

### P0-3：个人资料页改成真正稳定表单

必须修改：

1. 头像选择后立即显示：
   - 文件名
   - 文件大小
   - 格式校验结果
2. 头像上传按钮状态必须在头像表单内显示：
   - 待选择
   - 可上传
   - 上传中
   - 上传成功
   - 上传失败
3. 上传头像时不得整页重渲染清空 file input。
4. 保存个人资料、修改密码也必须用表单内状态，不只依赖顶部状态栏。
5. 密码错误必须显示在密码表单旁边。
6. 输入过程中收到后台录音状态，不得影响资料页。

验收：

1. 打开个人资料页，点击每个输入框都能获得焦点。
2. 输入 `中文 abc123` 后内容不丢失。
3. 选择头像后能看到文件名。
4. 点击上传头像后按钮变为“上传中”，成功后头像立即刷新。
5. 旧密码错误时，密码区域显示错误，不影响其它输入。
6. 保存成功后刷新侧栏，资料仍保留。

### P0-4：网页候选录音生成流程必须有卡片级状态机

当前“生成纪要”按钮不能再只是一个普通按钮。

候选卡片状态必须至少包含：

1. `idle`：可生成。
2. `reading`：正在读取网页录音。
3. `read_failed`：读取失败。
4. `creating_record`：正在创建记录。
5. `uploading`：正在上传到后端。
6. `processing`：已进入后端处理。
7. `failed`：流程失败。

每个状态必须显示在候选卡片内。

必须修改：

1. `uploadCandidate(index)` 开始后，立刻写入 `appState.candidateJobs[index]`。
2. `readCandidateBlob(candidate)` 使用独立超时，例如 20 秒。
3. 读取失败时，卡片内显示：
   - 失败原因
   - “改用手动上传”
   - “复制诊断”
   - “重新读取”
4. 创建记录成功但上传失败时，要显示已创建记录 ID，并允许重试上传或删除这条空记录。
5. 成功进入后端处理后，直接跳转详情页并显示状态条。

验收：

1. 点击“生成纪要”后 300ms 内按钮文字或卡片状态发生变化。
2. Plaud 录音地址读取失败时，卡片内明确显示失败原因。
3. 失败后“首页 / 上传 / 历史 / 配置”仍可点击。
4. 失败后可点“手动上传”进入上传页。
5. 不允许用户看到按钮点完完全没变化。

### P0-5：`runBusy()` 改成 action-scoped，不再默认整页重渲染

必须修改：

1. 引入 `appState.activeAction` 或 `appState.actionStates`。
2. `runBusy()` 支持 action key：

```js
await runAction('avatar.upload', async () => {})
await runAction(`candidate.${index}.upload`, async () => {})
```

3. 不再在每个动作开始前默认 `render()` 全页。
4. 对必须刷新 UI 的动作，只刷新局部状态或在动作结束后统一刷新。
5. 文件输入类动作不得在读取文件前重渲染。

验收：

1. 上传头像过程中 file input 不被提前清空。
2. 点击生成纪要时候选标题输入值不丢失。
3. 一个 action busy 不会把所有页面按钮都变成不可预测状态。
4. 多个异步任务结束顺序不同，也不会把当前页面回滚到旧状态。

### P0-6：加入侧栏前端错误边界和诊断

必须新增：

1. `window.addEventListener('error', ...)`
2. `window.addEventListener('unhandledrejection', ...)`
3. `reportClientError(error, context)`
4. `appState.clientErrors` 最近 20 条。
5. 写入 `chrome.storage.local.clientErrors`。
6. UI 顶部出现“插件前端异常”面板。
7. 面板提供：
   - 错误摘要
   - 发生时间
   - 当前 view
   - 最近 action
   - 复制诊断
   - 清除错误

验收：

1. 人为抛出异常后，侧栏不白屏。
2. 错误信息可见并可复制。
3. 重新打开侧栏后还能看到最近错误。
4. 插件需要重新加载时，界面能提示“扩展上下文已失效，请重新打开侧栏或重新加载扩展”。

## 8. 不允许的修法

1. 不允许只把顶部状态文案改得更明显，却不修 `body[data-view]` 误绑定。
2. 不允许继续用 `[data-view]` 全局选择器绑定导航。
3. 不允许头像上传开始前整页重渲染导致文件输入丢失。
4. 不允许网页录音读取失败只显示顶部状态。
5. 不允许一个按钮异常导致全局导航失效。
6. 不允许只用 HTML 字符串单元测试证明可用，必须有真实 DOM 点击测试。

## 9. 建议执行顺序

1. 先修 `body[data-view]` 误绑定。
2. 补真实 DOM 回归测试，证明输入框可点击。
3. 重构 `runBusy()` 为 action-scoped。
4. 修头像上传的局部状态。
5. 修候选录音生成卡片状态机和读取超时。
6. 增加前端错误边界和诊断面板。
7. 补完整 E2E 连续操作测试。
8. 真实 Chrome 扩展侧栏人工验收。

## 10. 完成定义

只有同时满足以下条件，才可以认为二次修复完成：

1. 个人资料页所有输入框真实可点击、可输入。
2. 头像选择、上传、失败、成功都有局部状态。
3. 点击“生成纪要”必有即时状态变化。
4. Plaud 直读失败有明确错误和手动上传兜底。
5. 任一动作失败后，其它导航和按钮仍可用。
6. 不再需要通过扩展管理页重新加载插件恢复普通交互。
7. 前端异常能在侧栏中看到并复制诊断。
8. 自动化测试覆盖真实 DOM 连续点击。
9. `npm run check && npm test` 通过。
10. 真实 Chrome 扩展侧栏人工验收通过。
