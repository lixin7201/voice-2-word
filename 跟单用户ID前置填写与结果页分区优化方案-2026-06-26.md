# 跟单用户 ID 前置填写与结果页分区优化方案

日期：2026-06-26
状态：待开发
建议版本：0.1.9

## 先回答当前三个问题

### 1. 用户 ID 输入不应该放在结果页

当前 0.1.8 的问题是：用户 ID 输入出现在录音详情、跟单信息或历史补填位置，这对“事后整理资料”有用，但不符合电话跟进场景。

真实工作流应该是：

1. 用户先在监听页选择“跟单信息”和跟单类型。
2. 用户打一通电话。
3. 插件监听到新录音后，立刻在这条新录音候选卡片上弹出或展开“用户 ID”输入框。
4. 用户填入客户/用户 ID 后点击“生成跟单”。
5. 系统马上提交录音、逐字稿、跟单任务，用户可以继续拨打下一通电话。
6. 处理结果完成后，用户后面回到历史页，直接搜索用户 ID 找到录音和跟单信息。

所以，用户 ID 是“这通电话的绑定字段”，不是“AI 生成结果出来以后再补的字段”。保存时机也应该提前到创建录音记录时，而不是等总结完成后再保存。

### 2. 跟单信息出现在“会议概览”里，是当前 UI 语义错位

现在代码里把“会议概览”当成了通用结果区域：只要记录里有跟单类型或跟单表单，就在 `renderSummaryPanel(record)` 里显示 `renderFollowupResultPanel(record)`。

这导致用户看到的是：

- 明明选择的是“跟单信息”，结果却出现在“会议概览”。
- 用户会自然理解为：会议概览 = 会议纪要结果。
- 跟单信息被塞进会议概览后，页面语义变得不可信。

正确逻辑应该是：

- “会议概览”只展示会议纪要、要点、待办、思维导图等会议类结果。
- “跟单信息”只展示跟进记录、客户沟通、商机整理、字段化跟单内容。
- 如果这条录音是跟单模式生成的，默认进入“跟单信息”页，而不是“会议概览”页。

### 3. 顶部“跟单/备注”现在职责不清，应该拆清楚

当前“跟单/备注”更像一个编辑入口：里面有跟单字段编辑、备注等内容。但因为跟单结果已经被放到了“会议概览”，用户会困惑：

- 会议概览里已经有跟单信息了，那“跟单/备注”是什么？
- 哪个地方才是正式的跟单结果？
- 哪个地方可以编辑？
- 备注和跟单是不是同一个东西？

建议调整为：

- 跟单模式下，顶部主结果页显示为“跟单信息”，里面展示 AI 生成的跟单结果、用户 ID、跟单类型、字段卡片。
- “备注”单独作为人工备注区，只放人工补充内容。
- 跟单字段编辑可以放在“跟单信息”页内，使用“编辑跟单字段”按钮或折叠区域，不再单独叫“跟单/备注”。

一句话：跟单结果归“跟单信息”，人工补充归“备注”，会议内容归“会议概览”。

## 核心目标

1. 监听到新录音时，如果当前选择了“跟单信息”，立即让用户填写用户 ID。
2. 用户提交录音后，可以马上进入下一通电话，不等待转写和总结结果。
3. 用户 ID 在创建录音记录时保存，后续历史页可直接搜索。
4. 跟单结果不再显示在“会议概览”里，改为独立的“跟单信息”结果区。
5. 顶部 tab 命名和内容一致，减少“点了但不知道在哪里”的感觉。
6. 历史搜索支持一个输入框同时搜索用户 ID 和关键词。

## 不做的事情

本次先不做完整客户管理系统，不新增客户表、客户画像、CRM 同步。

本次先不改 LLM 模型优先级，也不调整 sub2api、easyai、kimi 等模型配置。

本次先不做强制校验用户 ID，因为实际电话场景里可能存在一时找不到 ID 的情况。推荐做法是：跟单模式下默认展示并聚焦用户 ID 输入框，但允许用户留空继续生成，后续仍可补填。

## 优化后的监听流程

### 会议纪要模式

1. 用户选择“会议纪要”。
2. 插件监听到新录音。
3. 候选卡片显示录音标题、来源、时长、文件信息。
4. 按钮显示“生成纪要”。
5. 点击后进入转写和会议纪要生成。

会议纪要模式不显示用户 ID 输入框。

### 跟单信息模式

1. 用户选择“跟单信息”。
2. 用户选择跟单类型，例如“招聘跟单”。
3. 插件监听到新录音。
4. 候选卡片立刻显示：

   - 录音标题
   - 跟单类型
   - 用户 ID 输入框
   - “生成跟单”按钮

5. 用户填入用户 ID。
6. 点击“生成跟单”。
7. 系统创建录音记录时保存 `externalUserId`。
8. 详情页、跟单信息页、历史卡片里的用户 ID 自动使用同一个 `externalUserId`，不需要用户再填一次。
9. 录音卡片提示“已提交，可继续下一通电话”。
10. 转写和 AI 跟单在后台继续执行。
11. 后续用户在历史页搜索用户 ID，即可找到对应录音和跟单信息。

## 历史搜索建议

搜索框不需要拆成“用户 ID 搜索”和“关键词搜索”两个入口。建议保持一个搜索框，用户输入任意内容后，系统同时匹配：

- 用户 ID：`externalUserId`
- 录音标题：`title`
- 原始文件名：`originalFileName`
- 网页标题或来源：`sourcePageTitle`
- 负责人或员工名
- 跟单类型
- 跟单字段内容
- 跟单正文
- 会议纪要正文
- 逐字稿正文

第一版可以使用普通关键词包含匹配，不做复杂分词和高级检索。办公室录音数据量不大时，这样最直接，也最容易维护。

搜索结果展示建议：

- 命中用户 ID 时，在卡片上显示“用户 ID：xxx”。
- 命中跟单关键词时，在卡片上显示“跟单信息命中”。
- 命中逐字稿或会议纪要时，可以先只返回结果，不必第一版就做高亮摘要。

## 页面结构建议

### 会议纪要记录

顶部 tab 建议为：

- 会议概览
- 会议逐字稿
- 思维导图
- 备注
- 下载

默认打开“会议概览”。

### 跟单信息记录

顶部 tab 建议为：

- 跟单信息
- 逐字稿
- 思维导图
- 备注
- 下载

默认打开“跟单信息”。

“跟单信息”页展示：

- 用户 ID
- 跟单类型
- AI 生成的跟单阶段、建议标签、状态
- 字段化跟单信息
- 跟单正文
- 编辑跟单字段入口

其中“用户 ID”必须来自 `record.externalUserId`。如果用户在监听候选卡片里已经填写过，这里直接自动回显；如果当时留空，这里才允许后续补填。

“备注”页只展示人工备注，不再混放跟单结果。

## 需要修改的代码

### 1. `sidepanel.js`：监听候选卡片增加用户 ID 输入

涉及入口：

- `renderCandidateCard(candidate, index, primary)`，当前在约第 492 行。
- `renderCandidateAction(candidate, index, primary)`，当前负责候选卡片按钮。
- `uploadCandidate(index, options = {})`，当前在约第 3299 行。
- `renderProcessingPicker(prefix, selectedTemplate, selectedFollowup)`，当前在约第 2353 行。
- `setProcessingFollowupChoice(prefix, followupType)`，当前在约第 3638 行。

建议新增逻辑：

```js
function isCaptureFollowupMode() {
  const followupType = document.getElementById('capture-followup')?.value || currentFollowupType();
  return normalizeFollowupTypeUi(followupType, currentTemplateType()) !== 'none';
}
```

候选卡片渲染时：

```js
${isCaptureFollowupMode() ? renderCandidateExternalUserIdInput(candidate, index) : ''}
```

输入框建议文案：

- label：用户 ID
- placeholder：输入客户/用户 ID，方便稍后搜索
- hint：提交后可继续下一通，处理完成后可在历史中搜索该 ID

按钮文案建议：

- 会议纪要模式：生成纪要
- 跟单信息模式：生成跟单

### 2. `sidepanel.js`：上传候选录音时保存用户 ID

当前 `uploadCandidate(index, options = {})` 创建记录时会组装：

```js
const createBody = {
  sourceType: 'web_capture',
  sourcePageUrl: candidate.pageUrl || '',
  sourcePageTitle: candidate.pageTitle || '',
  title,
  titleSource,
  templateType,
  followupType,
  candidateUrl: candidate.url,
  forceDuplicate: Boolean(options.forceDuplicate),
};
```

跟单模式下需要加入：

```js
const externalUserId = readCandidateExternalUserId(candidate, index);

const createBody = {
  ...,
  externalUserId,
};
```

注意点：

- 用户 ID 需要在 `createRecord(createBody)` 前读取。
- 监听候选卡片里的用户 ID 是主来源；提交成功后，详情页下方的用户 ID 输入框必须自动回显这个值。
- 后续详情页如果手动修改用户 ID，只更新同一个 `record.externalUserId`，不能再出现“监听页一个 ID、详情页另一个 ID”的双字段。
- 不建议用纯 index 作为长期 key，因为候选列表可能刷新或重排。
- 建议用 `candidate.url`、`candidate.pageUrl`、`candidate.detectedAt` 组合一个稳定 key。
- 提交成功后清理这条候选录音对应的临时输入状态。

### 3. `sidepanel.js`：结果页 tab 需要按记录类型分区

涉及入口：

- `detailTabItems(record = {})`，当前在约第 905 行。
- `renderDetailTab(record)`，当前在约第 949 行。
- `renderSummaryPanel(record)`，当前在约第 1317 行。
- `renderFollowupResultPanel(record)`，当前在约第 1375 行。

当前问题逻辑类似：

```js
const showFollowup = normalizeFollowupTypeUi(record.followupType, record.templateType) !== 'none' || record.followupForm;
return showFollowup ? renderFollowupResultPanel(record) : renderMeetingResultPanel(record);
```

建议改为：

```js
function isFollowupRecord(record) {
  return normalizeFollowupTypeUi(record.followupType, record.templateType) !== 'none' || Boolean(record.followupForm);
}
```

tab 规则：

```js
function detailTabItems(record = {}) {
  if (isFollowupRecord(record)) {
    return [
      { id: 'followup', label: '跟单信息' },
      { id: 'transcript', label: '逐字稿' },
      { id: 'mind_map', label: '思维导图' },
      { id: 'notes', label: '备注' },
      { id: 'downloads', label: '下载' },
    ];
  }

  return [
    { id: 'summary', label: '会议概览' },
    { id: 'transcript', label: '会议逐字稿' },
    { id: 'mind_map', label: '思维导图' },
    { id: 'notes', label: '备注' },
    { id: 'downloads', label: '下载' },
  ];
}
```

渲染规则：

- `summary` 只渲染会议概览。
- `followup` 渲染跟单信息。
- `notes` 只渲染人工备注。

如果为了少改代码，可以保留内部 tab id，但对用户可见的 label 和渲染内容必须分开，不能再让“会议概览”显示跟单信息。

### 4. `sidepanel.js`：跟单编辑入口收敛

当前 `renderFollowupResultPanel(record)` 里已有“编辑跟单字段”相关内容，而顶部又有“跟单/备注”tab，会形成重复。

建议保留一个入口：

- 跟单信息页内：显示结果 + 编辑字段按钮。
- 备注页：只显示人工备注。

这样用户认知是：

- 看结果：跟单信息
- 改跟单字段：跟单信息里的编辑按钮
- 写人工补充：备注

### 5. `server/app.js`：保留用户 ID 保存与搜索

当前服务端已经有相关能力：

- 创建记录时保存 `external_user_id`，当前在约第 383 行。
- 更新记录时保存 `externalUserId`，当前在约第 436 行。
- 序列化返回 `externalUserId`，当前在约第 1165 行。
- 历史搜索会搜索 `external_user_id`，当前在约第 1435 行。

本次服务端重点不是大改，而是确认前端创建录音时把用户 ID 带进去。

需要补充检查：

- `POST /api/records` 接收到 `externalUserId` 后能保存。
- 历史搜索框输入用户 ID 能命中。
- 历史卡片或详情页能展示用户 ID。

需要新增关键词搜索范围：

- 记录基础字段：标题、原始文件名、网页标题、员工名、用户 ID。
- AI 结果字段：会议纪要、跟单正文、跟单字段。
- 逐字稿字段：如果当前列表接口已经能拿到 transcript，可直接匹配；如果没有，为了少改代码，第一版可以先匹配已有的 summary/followup 内容，逐字稿全文搜索放到下一版。

推荐第一版先扩展当前 `haystack` 拼接字段，不新增搜索服务、不引入分词依赖。

### 6. `server/lib/templates.js`：不建议本次大改提示词

当前 `buildSummaryPrompt(record, transcriptText, context)` 会在跟单模式下要求模型返回：

- `summaryMarkdown`
- `followupMarkdown`
- `followupFields`
- `followupStage`
- `suggestedTag`
- `statusLabel`

这可以先不大改。真正让用户困惑的是前端把跟单结果显示在“会议概览”里。

后续如果要进一步严格区分，可以再把模型输出拆成：

- 会议纪要模式：只要求会议纪要字段。
- 跟单信息模式：只要求跟单字段，`summaryMarkdown` 可以作为内部兼容字段，不作为“会议概览”展示。

本轮建议先修 UI 语义和保存时机。

## 最终判断

这不是简单增加一个输入框，而是要把“电话跟单”的主流程前置：

- 用户 ID 在录音被监听到时就绑定。
- AI 在后台慢慢处理。
- 用户继续打电话。
- 后面按用户 ID 回查。

同时要把结果页的概念改清楚：

- 会议概览只属于会议纪要。
- 跟单信息只属于跟单结果。
- 备注只属于人工补充。

按这个方向改，办公室同事第一次使用时会更容易理解：监听到录音、填用户 ID、生成跟单、继续下一通，后面搜索 ID 回看结果。
