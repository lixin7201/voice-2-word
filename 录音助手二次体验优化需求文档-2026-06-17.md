# 录音助手二次体验优化需求文档

日期：2026-06-17

## 1. 背景

本轮来自真实 Chrome 插件侧栏测试反馈。

已经确认改善：

1. 个人资料可编辑，头像可上传，输入框可正常输入。
2. 手动录音上传、识别、总结链路已可用。
3. Plaud 页面录音采集并生成总结已可用。

仍需优化：

1. 详情页把会议纪要、逐字稿、概览等内容堆在同一长页面里，查看成本高。
2. 历史记录里有大量测试数据，缺少选择删除能力，也没有同步删除存储桶对象。
3. 会议概览里没有显示思维导图；PDF 导出没有按原需求一次性导出“总结卡片 + 文字纪要 + 思维导图”。
4. 有的录音上传或进入识别很慢，状态不够透明；播放按钮点击后无法播放或缺少反馈。
5. 需要明确“最长能识别多长的录音”。

## 2. 本次只读排查结论

### 2.1 阿里云非实时语音识别限制

参考文档：

https://help.aliyun.com/zh/model-studio/non-realtime-speech-recognition-user-guide

官方限制与当前产品关系：

1. 非实时语音识别支持单个音视频文件异步转写，官方文档写明“单个时长不超过 12 小时、体积不超过 2GB”。
2. `qwen3-asr-flash-filetrans` 也标明支持最长 12 小时录音，但只接受公网音频文件 URL，不支持直接上传本地文件。
3. 同步调用 `fun-asr-flash-2026-06-15` 只适用于 5 分钟以内音频。
4. 查询任务接口默认 20 QPS，最高可扩容到 100 QPS。
5. 识别结果的 `transcription_url` 默认 24 小时有效，服务端必须及时下载并落盘保存。

当前插件后端状态：

1. 当前使用 `fun-asr` 非实时接口，代码在 `server/lib/dashscope.js`。
2. 当前服务端 DashScope 轮询默认超时为 20 分钟：`dashscopeTimeoutMs || 20 * 60 * 1000`。
3. 前端上传提示写的是“单个文件 ≤ 200MB”，但服务端没有真正校验 200MB，也没有校验 12 小时/2GB。
4. 真实 DashScope 识别需要 Cloudflare R2，因为 DashScope 需要公网可访问 URL。

产品结论：

1. API 理论上限：12 小时、2GB。
2. 当前插件不应宣称“无限制”。
3. 当前插件如果不改后台轮询/恢复策略，长录音可能在 20 分钟轮询超时处失败。
4. 下一版应明确显示：推荐录音长度、最大可提交限制、预计处理时间、超时后的恢复方式。

### 2.2 思维导图未显示的直接原因

最新真实记录中，`summaries.mind_map_json` 并非空，但结构是：

```json
{
  "topic": "家具广告合作沟通",
  "children": []
}
```

当前前端和导出代码只认：

```json
{
  "title": "录音思维导图",
  "center": "中心主题",
  "branches": []
}
```

因此：

1. 前端 `renderMindMap()` 判断 `mindMap.branches.length`，遇到 `{ topic, children }` 会直接不显示。
2. 导出 `renderMindMap()` 同样只认 `branches`，PDF 中会变成“暂无思维导图”。
3. 这不是“完全没有生成”，而是模型返回结构和产品内部结构不一致。

涉及位置：

1. `sidepanel.js`：`renderMindMap(rawMindMap)`。
2. `server/lib/exporters.js`：`renderMindMap(mindMap, markdown)`、`renderMindMapSvg(record, mindMap)`。
3. `server/lib/llm.js`：模型输出解析没有做 mind map 结构归一化。
4. `server/lib/templates.js`：本地兜底结构是 `branches`，与 LLM 返回不完全一致。

### 2.3 PDF 导出未满足原需求的原因

当前插件导出：

1. `summary` 的 PDF 会调用 `createSummaryPdfBuffer()`。
2. 该函数把总结卡片、文字重点、思维导图转成纯文本，再调用 `createPdfBuffer()`。
3. 即使思维导图结构正确，PDF 也不是 AI 大宜宾那种视觉化导出，只是文字化输出。

AI 大宜宾参考实现：

1. 代码位置：`/Volumes/AI-Code-Private/projects/ai-dayibin/src/lib/agents/audio/meeting-summary-export.ts`。
2. `buildSummaryExportHtml()` 把知识卡片、文字重点总结、思维导图组合进同一份 HTML。
3. `writeHtmlPdf()` 用 Playwright/Chrome 打印 A4 PDF，保留卡片和思维导图视觉结构。
4. `writeSummaryExportImages()` 还能把知识卡片和思维导图分别导出为 PNG，再用于 Markdown/DOCX。

迁移建议：

插件不要继续扩大纯文本 PDF；应引入 HTML -> PDF 的导出路径，最少先复用 AI 大宜宾的结构思路。

### 2.4 播放按钮不可播放的可能原因

本地接口验证：

1. 最新记录 `/api/records/:id/audio` 可返回音频文件。
2. 支持 Range 请求，`Accept-Ranges: bytes` 正常。
3. 但 Plaud 来源文件返回的 `Content-Type` 是 `binary/octet-stream`。

风险：

1. 浏览器 `<audio>` 对 `binary/octet-stream` 不如 `audio/mpeg` 稳定。
2. Plaud 下载返回二进制类型时，后端保存了原 MIME，播放时没有按扩展名纠正。
3. 监听页候选卡片里的圆形播放图标只是视觉图标，不是真正的播放器按钮；用户点击它不会播放候选录音，也没有解释。

涉及位置：

1. `server/app.js`：`serveRecordAudio()` 使用 `record.mime_type || contentTypeForAudio(ext)`。
2. `sidepanel.js`：`renderCandidateCard()` 里的 `.candidate-play` 只是 `aria-hidden` 图标。
3. `sidepanel.js`：详情页真正播放器在 `renderTranscriptWorkspace()`，需要明确入口与错误提示。

### 2.5 处理慢的原因拆解

本地记录里同样是约 16.8 分钟、约 3.85MB 的录音，处理总耗时差异明显：

1. 有的约 118 秒完成。
2. 有的约 147 秒完成。
3. 有的约 468 秒完成，其中 DashScope 任务提交到逐字稿生成约 383 秒。

这说明慢不一定卡在上传，也可能卡在：

1. 浏览器从 Plaud URL 读取音频。
2. 上传到本地后端。
3. 上传到 R2 并生成公网 URL。
4. DashScope 排队/转写。
5. LLM 生成总结。

当前问题是用户只看到“已进入后端处理”，无法知道卡在哪一段。

### 2.6 历史删除能力缺失

旧需求文档里写过：

`DELETE /api/records/:id`

但当前服务端没有实现该路由，前端历史页也没有删除按钮、批量选择、清理测试数据入口。

另外，当前 R2 工具只有 `putR2Object` 和 `presignR2Url`，没有删除对象能力。

## 3. 优化目标

### 3.1 用户目标

用户打开一条录音记录后，应能快速在以下栏目间切换，而不是被长页面拖着走：

1. 会议逐字稿
2. 会议概览/纪要
3. 思维导图
4. 跟单信息或备注
5. 下载/导出

每个栏目都应能展开/收起，长逐字稿默认不压住其它结果。

### 3.2 后台目标

1. 历史记录支持安全删除测试数据。
2. 删除记录时同步清理本地文件、导出文件、R2 音频、R2 导出文件。
3. 长录音处理可观测，可恢复，不让用户误以为卡死。
4. 明确录音长度/体积限制：不是无限制。

### 3.3 导出目标

PDF 导出必须一次性包含：

1. 总结卡片
2. 文字纪要
3. 思维导图

视觉效果应接近 AI 大宜宾，而不是纯文本拼接。

## 4. 详细需求

## P0-1：详情页改成横向栏目 + 可收起区域

### 当前问题

`renderDetail()` 每次渲染都写死：

```js
appState.detailTab = 'summary';
```

导致早前已有的 `renderDetailTab()` 实际未被使用。当前页面把概览和逐字稿放在同一工作台里，长逐字稿会让页面过长。

### 目标交互

详情页顶部增加横向栏目：

1. `会议逐字稿`
2. `会议概览`
3. `思维导图`
4. `跟单/备注`
5. `下载`

栏目要求：

1. 默认打开 `会议概览`。
2. 用户点击栏目后只切换内容区，不重置标题编辑、搜索词、说话人编辑等局部状态。
3. 每个栏目右上角提供“收起/展开”。
4. 逐字稿默认显示播放器 + 搜索框 + 前 80 段，继续沿用“显示更多”。
5. 思维导图栏目如果没有数据，要展示“已生成数据结构异常/暂无思维导图”的具体原因。

### 实施动作

1. 修改 `sidepanel.js`：
   - 删除 `renderDetail()` 中强制 `appState.detailTab = 'summary'`。
   - 新增 `renderDetailTabs(record)`。
   - 复用现有 `renderDetailTab(record)`，但把 tab 名称调整成用户语言。
   - 新增 `appState.collapsedPanels`，记录每个栏目是否收起。
   - `bindCurrentView()` 增加 `button[data-detail-tab]` 和 `button[data-action="toggle-detail-panel"]`。
2. 修改 `sidepanel.css`：
   - 新增 sticky/横向滚动的 `.detail-tabs`。
   - 移动端保持一行横向滑动，不换成下拉。
3. 更新测试：
   - 打开详情页默认为会议概览。
   - 点击逐字稿后不重置页面。
   - 收起概览后再次渲染仍保持收起状态。

### 验收标准

1. 16 分钟以上录音详情页不再出现“一个页面到底”的压迫感。
2. 用户能一眼看到横向栏目。
3. 逐字稿、概览、思维导图可以独立查看。
4. 切换栏目不会造成输入框失焦或回到顶部。

## P0-2：修复思维导图结构兼容

### 当前问题

LLM 返回 `{ topic, children }` 时，前端和导出都不显示。

### 目标

系统内部统一使用：

```json
{
  "title": "录音思维导图",
  "center": "中心主题",
  "branches": [
    {
      "id": "branch-1",
      "title": "分支",
      "summary": "",
      "tone": "blue",
      "children": []
    }
  ]
}
```

但兼容旧结构：

1. `{ topic, children }`
2. `{ title, children }`
3. `{ center, nodes }`
4. 纯数组 children

### 实施动作

1. 新增 `normalizeMindMap(value, fallbackTitle)`：
   - 放在 `server/lib/templates.js` 或新建 `server/lib/mind-map.js`。
   - 前端可另建轻量同名归一化函数，或由服务端序列化时统一处理。
2. 服务端写入 summaries 前统一归一化：
   - `summarizeExistingRecord()`
   - dev fake/local summary 写入路径
   - speaker alias 同步后也应保持结构。
3. `serializeRecord()` 返回前可再次兜底归一化，保证旧记录无需重新生成也能显示。
4. 导出函数只接收归一化后的结构。

### 验收标准

1. 当前记录 `家具广告合作方案沟通` 能显示思维导图。
2. 不重新生成总结也能显示旧的 `{ topic, children }` 思维导图。
3. PDF、DOCX、SVG、Markdown 中都不再出现“暂无思维导图”，除非确实没有数据。

## P0-3：PDF 导出改为视觉化完整导出

### 当前问题

当前 `summary` PDF 是纯文本 PDF，不能达到“总结卡片 + 文字纪要 + 思维导图一次性导出”的体验。

### 参考实现

参考 AI 大宜宾：

1. `meeting-summary-export.ts`
2. `buildSummaryExportHtml()`
3. `writeHtmlPdf()`
4. `writeSummaryExportImages()`

### 目标导出

下载“总结 PDF”时，PDF 内容顺序必须是：

1. 封面：录音标题、员工、部门、创建时间、来源。
2. 总结卡片：保留卡片布局和颜色。
3. 文字纪要：Markdown 渲染成标题、列表、段落。
4. 思维导图：保留中心主题、分支、节点。
5. 可选：跟单信息或备注，放在后面，不干扰会议纪要主结构。

### 实施动作

1. 在 `server/lib/exporters.js` 中新增 HTML 导出路径：
   - `buildSummaryExportHtml(record, store)`
   - `renderOverviewCardHtml(card)`
   - `renderMindMapHtml(mindMap)`
   - `markdownToHtml(markdown)`
2. 新增 `writeHtmlPdf(buffer/path)`：
   - 优先使用本机 Chrome：`/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`。
   - 如果后续引入 Playwright，则按 AI 大宜宾实现方式封装。
3. 保留现有纯文本 PDF 作为 fallback：
   - Chrome/Playwright 不可用时返回纯文本 PDF，但必须写入导出事件/错误提示。
4. DOCX 导出：
   - 可以继续使用 SVG/文本方式，但应确保思维导图结构兼容后能正常带入。

### 验收标准

1. 下载总结 PDF 后，肉眼能看到总结卡片、文字纪要、思维导图三块。
2. 当前记录导出的 PDF 不再只有纯文字。
3. PDF 首页不卡中文字体，不出现乱码。
4. 自动测试至少验证 PDF buffer 大小、包含三块标题、HTML 构建函数输出含三块 section。

## P0-4：修复录音播放 MIME 和候选播放误导

### 当前问题

1. Plaud 音频接口返回 `binary/octet-stream`，后端播放接口照原样输出。
2. 候选卡片圆形播放图标不可点击，容易让用户以为能播放。

### 目标

1. 详情页播放器可以播放 Plaud mp3。
2. 候选页不要用不可点击的播放图标误导。
3. 如果无法播放，要给出可读原因和“打开音频文件/下载后手动上传”的路径。

### 实施动作

1. 修改 `server/app.js` 的 `serveRecordAudio()`：
   - 如果 `record.mime_type` 是 `binary/octet-stream`、`application/octet-stream`、空值，则用扩展名推断。
   - mp3 返回 `audio/mpeg`。
   - m4a/mp4 返回 `audio/mp4` 或 `video/mp4`。
2. 修改 `sidepanel.js`：
   - 监听页候选卡片把 `.candidate-play` 改成状态图标，或加 tooltip/文字“候选录音，点击生成纪要，不在这里播放”。
   - 详情页播放器增加 `error` 事件监听，显示“浏览器无法播放该格式，可下载后播放/重新上传”。
3. 增加测试：
   - Plaud `binary/octet-stream + .mp3` 返回 `audio/mpeg`。
   - audio Range 请求仍为 206。

### 验收标准

1. 当前 Plaud mp3 详情页可播放。
2. 点击逐字稿时间戳能跳转到对应播放位置。
3. 播放失败时有明确错误提示，不是静默无效。

## P0-5：处理进度可观测 + 长录音策略

### 当前问题

用户只能看到“已进入后端处理”，不知道卡在上传、R2、DashScope 还是总结。

### 目标状态

每条记录展示清晰阶段：

1. 浏览器读取网页录音
2. 上传到后端
3. 后端保存文件
4. 上传到 R2
5. 已提交 DashScope，显示 task_id
6. DashScope 转写中，显示已等待时长
7. 逐字稿已生成，正在总结
8. 总结完成
9. 失败，可重试

### 长录音限制

显示规则：

1. 官方上限：12 小时、2GB。
2. 产品推荐：2 小时以内体验最佳。
3. 超过 2 小时但不超过 12 小时：允许提交，但提示“预计处理时间较长，可离开页面稍后回来”。
4. 超过 12 小时或 2GB：前端和后端都拒绝，并提示切分录音。

后端策略：

1. `DASHSCOPE_TIMEOUT_MS` 默认不应再是 20 分钟；如果继续同步等待，至少提高到 13 小时不现实。
2. 更合理做法：提交 DashScope 后立即返回，后台定时恢复/轮询。
3. 已有 `recoverInterruptedRecords()`，应加强为“可恢复任务队列”：
   - 记录 `asr_task_id`。
   - 记录 `last_progress_at`。
   - 服务重启后继续查任务。
   - 用户点击详情页可手动刷新状态。

### 实施动作

1. 服务端 `processUploadedRecord()` 不要长时间阻塞 HTTP 请求。
2. 上传接口返回后，前端立即进入详情页，轮询记录状态。
3. `record_processing_events` 序列化到详情页并展示时间线。
4. 增加“复制处理诊断”按钮：
   - record id
   - file size
   - duration
   - r2 key 是否存在
   - asr task id
   - last progress
   - latest error
5. 增加长时间无进展提示：
   - 5 分钟：提示“仍在转写，可稍后查看”。
   - 20 分钟：提示“长录音或服务排队中，已保存任务 ID”。
   - 超过配置阈值：显示“重新查询 DashScope 状态”。

### 验收标准

1. 上传 10 分钟录音时，用户能看到每个阶段。
2. 上传 1 小时模拟录音时，HTTP 请求不长时间卡住。
3. 服务重启后，已有 `asr_task_id` 的记录能恢复查询。
4. 失败时用户能复制诊断信息。

## P1-1：历史记录删除与测试数据清理

### 目标

历史记录页支持：

1. 单条删除。
2. 批量选择删除。
3. 只看测试数据/按来源筛选。

建议提供两种：

1. 归档：默认动作，不展示在普通历史列表，但保留数据库和文件。
2. 彻底删除：管理员可用，删除数据库记录、本地文件、R2 对象、导出文件。

1. `DELETE /api/records/:id?mode=archive|purge`
2. `POST /api/records/bulk-delete`

权限：

1. 普通员工只能归档自己的记录。
2. 管理员可以归档或彻底删除所有记录。
3. 彻底删除必须写审计日志。

### R2 删除

在 `server/lib/r2.js` 增加：

1. `deleteR2Object(config, key)`
2. `deleteR2Objects(config, keys)`

彻底删除应清理：

1. `audio_records.r2_key`
2. `export_files.r2_key`
3. 本地 `uploads/<recordId>.<ext>`
4. 本地 `exports/<recordId>-*`
5. `transcripts`
6. `summaries`
7. `followup_forms`
8. `record_notes`
9. `record_processing_events`

### 验收标准

1. 删除一条测试记录后，历史列表消失。
2. 本地 uploads/exports 对应文件消失。
3. R2 配置存在时，R2 对象删除成功；失败时显示部分失败，不伪装成功。

## 7. 风险与注意事项

1. 不要在删除功能里默认物理删除；先归档，彻底删除必须二次确认。
2. R2 删除失败不能把数据库先删光，必须保留失败记录或审计。
3. 长录音不要靠一个 HTTP 请求一直等到结束，应拆成后台任务。
4. 视觉化 PDF 引入 Chrome/Playwright 后，要考虑部署机器是否有可用浏览器。
5. mind map 归一化要兼容旧数据，不能要求用户重新生成全部历史总结。
6. 不能承诺无限识别，公开话术应是：官方接口支持单文件最长 12 小时、最大 2GB；插件会按这个上限做校验，超长建议切分。

## 8. 对用户的简明说明

这个插件不是无限制识别。按阿里云当前文档，非实时识别单个文件最长 12 小时、最大 2GB。我们当前插件还需要补上明确的前后端校验、长任务后台恢复和进度展示。下一版改完后，用户看到的口径应是：

> 推荐 2 小时内录音直接处理；最长支持 12 小时、2GB 以内文件。长录音会进入后台慢慢识别，页面会显示当前卡在哪一步；超过上限请先拆分录音。
