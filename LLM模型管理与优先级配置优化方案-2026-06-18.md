## 0. 这次修正的关键结论

先把事实纠正清楚：

1. `sub2api` 不是一个待猜测的普通 OpenAI Chat 兼容模型。它已经在 AI 大宜宾项目里作为 `gpt-5.5` 的本机反代通道存在。
2. AI 大宜宾里的 `gpt-5.5` 不是只走 EasyAI。真实逻辑是：EasyAI GPT-5.5 优先，Sub2API GPT-5.5 兜底；在 agent 模型配置里两条通道可以同时保留。
3. AI 大宜宾的 `gpt-5.5` 调用协议是 OpenAI Responses，不是 Chat Completions。真实请求路径是 `POST {baseUrl}/responses`，其中默认 sub2api base 是 `http://127.0.0.1:8080/v1`。
4. 当前录音助手项目仍然只把 EasyAI/Kimi 当成两组写死配置，并且当前 LLM 调用层走的是 Chat Completions。它还没有按 AI 大宜宾的真实模型体系接入 `sub2api`。
5. 录音助手要做“管理员新增模型、输入 Base URL/API Key、测试通过后一键加入、可调整优先级”，不能只做一个简单 `baseUrl + apiKey + model` 表单，必须同时保存协议类型：`openai-responses`、`openai-chat`、`anthropic-messages`、`gemini-native`。

本文不记录任何真实 API Key。下面凡是密钥都只写“已配置/未配置”或字段名。

## 3. 当前录音助手的真实差距

当前录音助手代码位置：`/Users/lixin/AI code/voice to word`

| 模块 | 当前状态 | 与 AI 大宜宾真实模型体系的差距 |
|---|---|---|
| `server/lib/system-settings.js` | 总结模型配置写死为 EasyAI 和 Kimi 两组字段 | 没有模型池；没有 sub2api；没有协议类型；没有优先级列表 |
| `server/lib/llm.js` | `generateSummary` 组装 providers：EasyAI -> Kimi，统一调用 Chat Completions | GPT-5.5 应走 Responses；sub2api 也应走 Responses；Kimi 才走 Chat Completions |
| `server/app.js` | 健康检查只检查 easyai/kimi 必填字段 | 需要改成检查启用的模型池，并显示“至少一个可用模型” |
| 管理员配置页 | 只展示固定字段 | 需要模型列表、新增、测试、排序、启停、删除 |
| 密钥保存 | 已有 system_settings secret 字段不回显 | 可以复用密钥不回显规则，但需要支持多模型密钥记录 |

所以，录音助手不是“把 EasyAI 模型名改成 sub2api”就能完成。正确改法是：引入协议感知的模型池，然后把 EasyAI GPT-5.5、Sub2API GPT-5.5、Kimi K2.6 按真实协议接入。

## 4. 目标产品形态

管理员在“配置”页新增“总结模型池”。

管理员可完成：

1. 查看当前模型池：模型名、协议、Base URL、模型 ID、优先级、启用状态、最近测试、最近调用。
2. 新增模型：输入显示名称、协议类型、Base URL、API Key、Model。
3. 使用预设模板快速填入：
   - EasyAI GPT-5.5
   - AI 大宜宾 sub2api GPT-5.5
   - Kimi K2.6
   - OpenAI Responses 兼容
   - OpenAI Chat 兼容
   - Anthropic Messages 兼容
   - Gemini Native 兼容
4. 点击“测试连接”，后端真实调用一次模型。
5. 测试通过后保存或启用。
6. 调整优先级：上移、下移，或数字排序。
7. 设置失败后是否继续尝试下一个模型。
8. 查看每次总结实际命中的 provider、model、耗时和失败摘要。

普通员工、部门领导、老板不应看到 API Key 明文。普通员工不应看到模型配置入口。

### 5.1 字段规则

| 字段 | 规则 |
|---|---|
| `display_name` | 管理员可读名称，1 到 80 字 |
| `provider_key` | 小写字母、数字、短横线、下划线 |
| `channel_id` | 用于记录真实命中通道，例如 `easyai`、`sub2api`、`kimi` |
| `protocol` | 必须是 `openai-responses`、`openai-chat`、`anthropic-messages`、`gemini-native` 之一 |
| `base_url` | 可为 `https://`；允许本机后端使用 `http://127.0.0.1` 或 `http://localhost`；不允许前端直接调用 |
| `endpoint_path` | 可为空，由协议默认生成；特殊模型可覆盖 |
| `api_key` | 后端保存，前端永不回显明文 |
| `request_model` | 实际请求里的 `model` |
| `priority` | 数字越小越先调用 |
| `enabled` | 停用后不参与总结 |
| `allow_fallback` | 当前模型失败后是否继续尝试下一个模型 |
| `timeout_ms` | 默认 120000，范围 10000 到 300000 |
| `reasoning_effort` | 仅 `openai-responses` 的 GPT-5.5 使用，允许 `low/medium/high/xhigh` |

## 7. 后端接口设计

所有模型管理接口必须只允许管理员或老板访问。不能只靠前端隐藏按钮。

### 7.1 获取模型池

```text
GET /api/admin/llm-providers
```

返回规则：

1. 不返回 `api_key` 明文。
2. 返回 `configured: true/false`。
3. 返回 `maskedApiKey`，例如 `sk-***abcd` 或 `已保存`。
4. 返回排序后的模型列表。

### 7.2 测试模型

```text
POST /api/admin/llm-providers/test
```

请求：

```json
{
  "protocol": "openai-responses",
  "baseUrl": "http://127.0.0.1:8080/v1",
  "apiKey": "只在请求中传入，不回显",
  "requestModel": "gpt-5.5",
  "reasoningEffort": "high"
}
```

返回：

```json
{
  "ok": true,
  "message": "模型测试通过",
  "latencyMs": 1234,
  "provider": "sub2api",
  "requestModel": "gpt-5.5"
}
```

失败返回只能是安全摘要：

```json
{
  "ok": false,
  "message": "模型测试失败：HTTP 401，请检查 API Key。"
}
```

禁止返回完整 API Key、完整 Authorization header、完整上游堆栈。

### 7.3 新增模型

```text
POST /api/admin/llm-providers
```

保存前规则：

1. `protocol/baseUrl/apiKey/requestModel` 必填。
2. 新增前必须测试通过，或者请求体明确带 `forceSaveWithoutTest` 且只允许管理员使用。
3. `baseUrl` 自动去掉末尾 `/`。
4. 对 `openai-responses` 和 `openai-chat`，如果 base 没有 `/v1`，允许系统提示“是否自动补 `/v1`”，但不要盲目改第三方特殊地址。

### 7.4 更新模型

```text
PATCH /api/admin/llm-providers/:id
```

可更新字段：

```text
display_name
provider_key
channel_id
protocol
base_url
endpoint_path
api_key
request_model
priority
enabled
allow_fallback
timeout_ms
reasoning_effort
```

如果 `api_key` 空且未明确 `clearApiKey: true`，保留原 Key。

### 7.5 调整优先级

```text
POST /api/admin/llm-providers/reorder
```

请求：

```json
{
  "ids": ["llm_easyai_gpt55", "llm_sub2api_gpt55", "llm_kimi_k26"]
}
```

后端按数组顺序写入稳定 priority。

### 7.6 删除或停用模型

```text
DELETE /api/admin/llm-providers/:id
PATCH /api/admin/llm-providers/:id
```

建议默认“停用”，不要物理删除；物理删除需二次确认。

### 7.7 总结时实际调用模型

录音助手现有 `POST /api/records/:id/summarize` 不需要改用户入口，但内部要改成：

1. 读取启用的模型池。
2. 按 `priority ASC` 排序。
3. 逐个调用协议 adapter。
4. 成功后写入 `summaries.model_provider`、`summaries.model_name`、`summaries.model_error`。
5. 失败时记录每个模型的安全错误摘要。
6. 全部失败后，才回退本地模板，并把 `model_error` 写清楚。

## 8. 协议 Adapter 设计

### 8.1 `openai-responses`

用于：

1. EasyAI GPT-5.5
2. AI 大宜宾 sub2api GPT-5.5
3. Custom GPT-5.5 / Evolink direct GPT-5.5

请求：

```text
POST {baseUrl}/responses
Authorization: Bearer {apiKey}
Content-Type: application/json
```

Payload：

```json
{
  "model": "gpt-5.5",
  "instructions": "系统提示词",
  "input": [
    {
      "role": "user",
      "content": [
        {
          "type": "input_text",
          "text": "请根据以下逐字稿生成结构化总结..."
        }
      ]
    }
  ],
  "reasoning": {
    "effort": "high"
  }
}
```

解析优先级：

1. `output_text`
2. `output[].content[].text`
3. `output[].content[].output_text`
4. 兼容 fallback：`choices[0].message.content`

### 8.2 `openai-chat`

用于：

1. Kimi K2.6
2. 其他 Chat Completions 兼容模型

请求：

```text
POST {baseUrl}/chat/completions
Authorization: Bearer {apiKey}
Content-Type: application/json
```

Payload：

```json
{
  "model": "kimi-k2.6",
  "messages": [
    {
      "role": "system",
      "content": "你是大宜宾录音助手..."
    },
    {
      "role": "user",
      "content": "逐字稿..."
    }
  ],
  "temperature": 0.2
}
```

解析：`choices[0].message.content`。

### 8.3 `anthropic-messages`（二期）

用于 DeepSeek V4 Pro。

AI 大宜宾 catalog 显示该模型 endpoint 为 `/v1/messages`，但录音助手一期不建议先接，除非同时实现完整 Anthropic Messages payload、header 和响应解析。

### 8.4 `gemini-native`（二期）

用于 Gemini 3.1 Flash Lite 2 / Gemini 3.1 Pro。

AI 大宜宾 catalog 已有模型定义，但录音助手总结链路第一期不需要多模态原生接口。若接入，必须单独实现 `generateContent` payload 和响应解析。

## 11. 迁移策略

### 11.1 从旧配置迁移

当前录音助手旧配置：

```text
easyAiBaseUrl
easyAiApiKey
easyAiModel
kimiBaseUrl
kimiApiKey
kimiModel
```

迁移规则：

1. 如果 `easyAiApiKey` 存在，创建 `EasyAI GPT-5.5` provider。
2. 旧 `easyAiModel` 如果是 `gpt-5.5`，protocol 必须设为 `openai-responses`。
3. 如果 `kimiApiKey` 存在，创建 `Kimi K2.6` provider，protocol 设为 `openai-chat`。
4. 新增 `AI 大宜宾 sub2api - GPT-5.5` 模板，但不自动填 Key，除非管理员手动输入。
5. 迁移后保留旧配置字段至少一个版本，作为回滚保障。

### 11.2 启动时兼容

总结生成逻辑：

1. 优先读取 `llm_providers` 中启用且 key 已配置的模型。
2. 如果没有可用 provider，则读取旧 EasyAI/Kimi 配置组装临时 providers。
3. 如果仍然没有可用 provider，使用本地模板兜底。

这样能保证上线过程中不会因为模型池为空导致总结完全不可用。

## 12. 安全要求

1. API Key 只能保存在后端 store 或数据库，不能进入 Chrome 扩展源码、manifest、sidepanel JS 常量。
2. 所有 `/api/admin/llm-providers*` 接口必须校验管理员/老板权限。
3. 返回给前端的模型列表不能包含 `api_key` 明文。
4. 测试失败不能返回完整上游响应堆栈。
5. 日志里不能打印 Authorization header。
6. 审计日志要记录：谁新增模型、谁修改模型、谁测试模型、谁调整优先级、谁删除/停用模型。
7. `http://127.0.0.1`、`http://localhost` 只允许后端本机调用；不能让浏览器扩展直接用它。
8. Base URL 必须做白名单规则：
   - 允许 `https://*`
   - 允许后端本机 `http://127.0.0.1:*`
   - 允许后端本机 `http://localhost:*`
   - 不允许普通内网任意地址，除非管理员开启高级模式
9. 测试模型时设置超时，避免一个坏模型卡住总结任务。
10. 失败切换要记录模型级错误摘要，但不要暴露 Key。

## 17. 最终产品判断

要实现用户提出的“几个模型都可以上，甚至可以新增模型，输入 baseurl 和 api-key，点击测试，测试通过一键新增，并且可以调整优先级”，最稳妥的落地是：

1. 一期先支持 `openai-responses` 和 `openai-chat` 两类协议。
2. 先把 EasyAI GPT-5.5、AI 大宜宾 sub2api GPT-5.5、Kimi K2.6 接稳。
3. DeepSeek/Gemini 先作为二期模板存在，未实现 adapter 前不能启用。
4. 所有模型都由录音助手后端调用，Chrome 扩展只请求录音助手后端。
5. 管理员可以调整优先级，系统按优先级逐个失败切换。

这样既能准确复用 AI 大宜宾现有 sub2api 事实，也不会破坏录音助手已经实现的录音、上传、转写、总结、历史和导出功能。
