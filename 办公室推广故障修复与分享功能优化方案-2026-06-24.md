# 大宜宾录音助手办公室推广故障修复与分享功能优化方案

更新时间：2026-06-24

项目路径：

```text
/Users/lixin/AI code/voice to word
```

本文是给下一位 AI 或开发者直接执行的优化文档。本次先不修改业务代码，目标是把办公室真实推广中暴露的问题拆成可排查、可实现、可验收的任务。

## 0. 本次现场反馈

办公室正式推广后，第一个同事安装使用就遇到两个严重问题：

1. 逐字稿已生成，但页面提示 AI 总结失败。
2. 监听新录音失败，不管如何刷新页面，都无法监听到新的录音文件。

同时需要新增类似 Plaud 的分享能力：

1. 生成逐字稿、总结后，可以创建分享链接。
2. 分享时可以自由勾选：录音、逐字稿、总结。
3. 分享链接在办公室网络环境下可访问。
4. 被分享人可以在手机上打开后台页面，看文字、听录音。
5. 分享体验参考 Plaud 弹窗：勾选内容、复制链接、默认有效期，过期后失效。

## 1. 先给结论

### 1.1 总结失败不是同一类问题

截图里的提示是：

```text
AI 总结模型暂不可用，已保留逐字稿，可重试生成总结。
AI 总结未成功，逐字稿已保留。真实总结模型全部失败，系统仅生成临时模板。
```

这说明当前系统至少有一部分“质量闸门”已经生效：它没有把失败总结伪装成完成，而是保留逐字稿并提示重试。

因此，第一步不要立刻大改总结逻辑，而是先判断现场属于哪一种：

1. 如果同事安装的是旧插件或后端旧包：先升级到最新包，再复测。
2. 如果已经是最新包：重点查模型池、网络、额度、超时和后台配置。
3. 如果只有某条记录失败：复制该记录处理诊断，不能泛化成“整个插件坏了”。

当前仓库已具备相关基础：

1. `server/lib/summary-quality.js`：已有总结质量判定。
2. `server/app.js` 的 `persistSummaryResult()`：会把 `fallback_template` / `invalid` 降级为 `transcribed`。
3. `sidepanel.js` 的 `hasSummaryContent()` / `summaryIsUsable()`：不会把不可用总结当成可下载正式总结。

### 1.2 监听失败要按浏览器扩展问题处理

监听新录音失败通常不靠刷新页面解决。录音监听链路在 Chrome 扩展后台：

1. `background.js` 负责监听当前标签页的网络请求和页面媒体元素。
2. `sidepanel.js` 负责把候选录音展示在“监听”页。
3. 后端只有在查重、创建记录、上传文件时才参与。

所以现场要先区分：

1. 是没有发现候选录音。
2. 是发现了候选，但不能读取。
3. 是读取成功，但上传失败。
4. 是上传成功，但转写或总结失败。

这四类问题的修法不同，不能混成一个“刷新页面”动作。

### 1.3 分享功能要独立建“分享链接”体系

现有导出接口适合本人下载文件，不适合分享给手机端访问：

1. `POST /api/records/:id/export` 会生成导出文件。
2. 导出下载 token 默认只有短时间有效。
3. `/api/records/:id/audio` 当前依赖登录 token 或查询 access_token。

分享功能需要新增独立的只读公开访问体系：

1. 创建分享链接时存勾选范围。
2. 分享 token 独立于登录 token。
3. 手机打开 `/s/:token`，不需要安装插件。
4. 后端按 token、有效期、撤销状态、勾选范围返回内容。
5. 录音播放必须支持手机浏览器的 Range 请求。

## 2. 不做的范围

本次不要顺手做这些事：

1. 不做公网分享。第一期只支持办公室同一局域网。
2. 不做自动静默更新 Chrome 解压扩展。现有安装方式仍是下载 zip、解压、重新加载。
3. 不把分享链接变成协作编辑页。被分享人只读。
4. 不允许被分享人修改逐字稿、总结、跟单信息。
5. 不绕过现有员工权限创建分享。谁能创建分享，仍由记录权限控制。
6. 不把所有历史导出逻辑推翻重写。分享页可以复用已有记录、转写、总结数据。
7. 不把真实密钥、R2 key、模型错误细节暴露到分享页。
8. 不在这份文档里直接改业务代码。

## 3. P0 现场止血排查

### 3.1 第一个同事电脑必须采集的信息

在同事电脑上先记录：

| 项目 | 获取方式 | 期望 |
| --- | --- | --- |
| 插件版本 | 插件侧栏配置页“复制扩展诊断”，或 Chrome 扩展详情 | 应为最新发布包版本 |
| 后端地址 | 插件登录页或配置页 | 应指向 `http://lixindemac-studio.local:8127` 或当前真实局域网地址 |
| 后端健康 | 同事电脑浏览器打开 `/health` | `ok: true`，且能看到 `llmConfigured` / `dashscopeConfigured` |
| 安装来源 | 是否从 `/install` 下载 zip | 不应直接加载旧目录或旧压缩包 |
| 失败记录 ID | 详情页复制处理诊断 | 后续查 `audio_records`、`transcripts`、`summaries` |
| 监听诊断 | 监听页或配置页复制扩展诊断 | 要包含 `globalStatePhase`、`candidatesCount`、`lastScanError`、`extensionVersion` |
| Plaud 页面 | 录音详情页 URL 和是否点击播放 | 必须在监听开始后点击播放录音 |

如果没有这些信息，不要先猜结论。

### 3.2 后端健康检查

管理员电脑或同事电脑访问：

```text
http://lixindemac-studio.local:8127/health
http://lixindemac-studio.local:8127/api/runtime
http://lixindemac-studio.local:8127/api/extension/latest
```

必须确认：

1. 后端能从同事电脑访问。
2. `version` 是当前预期版本。
3. `latestExtension.version` 与发布包一致。
4. `latestExtension.minSupportedVersion` 不高于同事实际版本，或同事已按提示升级。
5. `llmConfigured` 为 `true`，否则总结失败是配置问题。

手机分享功能上线前，还要用手机浏览器访问同一个地址。如果 `.local` 在某些手机上不可用，优先在后台把 `publicBaseUrl` 设置为固定局域网 IP，例如：

```text
http://192.168.x.x:8127
```

不要假设所有手机都能稳定解析 `.local`。

### 3.3 总结失败排查顺序

按下面顺序查：

1. 看失败记录是否已有逐字稿。
   - 有逐字稿：转写链路成功，只查总结模型。
   - 无逐字稿：先查 DashScope、R2、上传。
2. 看 `summary.quality_status`。
   - `fallback_template`：真实总结模型全部失败。
   - `invalid`：模型返回为空或过短。
   - `low_information`：生成了，但信息量不足。
   - `ai_ok`：总结正常，前端展示可能有问题。
3. 看 `summary.provider_errors_json`。
   - 超时：调大该 provider 超时，或临时关闭慢模型。
   - 余额/额度不足：换 key 或充值。
   - 未配置：管理员后台补配置。
   - 地址不可达：检查办公室网络、sub2api、代理、端口。
4. 在管理员配置页逐个测试模型池。
5. 修复模型后，在记录详情页点“重新生成”。

验收标准：

1. 模型全部失败时，记录只能保持“已转写，待生成总结”，不能显示“完成”。
2. 真实模型恢复后，点击“重新生成”能从同一条逐字稿生成总结，不重新上传录音。
3. 总结成功后，下载总结、复制总结、跟单信息仍可用。

### 3.4 监听失败排查顺序

在 Plaud 或其他录音网页上按固定流程复测：

1. 打开录音详情页。
2. 打开插件侧栏。
3. 切到“监听”页。
4. 点击“识别当前页录音”。
5. 回到网页点击播放按钮，让浏览器真正发起音频请求。
6. 等待候选录音出现。

如果没有候选：

1. 点“复制扩展诊断”。
2. 记录 `globalStatePhase`、`candidatesCount`、`lastScanError`。
3. 打开 Chrome 扩展管理页，确认扩展未报错。
4. 不要反复刷新网页后直接说失败，因为刷新会清空当前 tab 的网络候选。

如果有候选但读取失败：

1. 点击候选卡片的“复制诊断”。
2. 看 `uploadable`、`unsupportedReason`、`urlHost`、`size`、`rangeSize`。
3. 如果是 blob、m3u8、低置信链接，走手动上传兜底。
4. 如果是 206 分片或大小异常，优先检查 `Content-Range` 和 `validateCandidateBlobComplete()`。

如果读取成功但上传失败：

1. 看是否已经创建记录 ID。
2. 能重试上传同一记录时，不要重新创建新记录。
3. 查 R2 配置、文件名、文件大小、后端 `/api/records/:id/upload`。

验收标准：

1. 正常 Plaud 录音详情页能在点击播放后出现候选。
2. 刷新页面后，点击“识别当前页录音”能重新建立监听，不需要重装插件。
3. 失败时必须能复制诊断，且诊断足够判断是“未发现候选 / 读取失败 / 上传失败”。

## 4. P1 监听可靠性优化

### 4.1 当前关键文件

1. `background.js`
   - `scanTab()`
   - `collectCandidatesFromTab()`
   - `rememberNetworkCandidate()`
   - `getDiagnostics()`
   - `setupTabLifecycleListeners()`
2. `sidepanel.js`
   - `startScan()`
   - `handleBackgroundState()`
   - `renderCapture()`
   - `copyCandidateDiagnostics()`
   - `shouldAutoOpenCapture()`
3. `server/background-capture.test.js`
4. `server/sidepanel-ui.test.js`

### 4.2 要解决的问题

1. 同事不知道“刷新页面”不会自动等于“重新监听”。
2. 当前监听状态可能停留在旧 tab、旧页面或旧候选上。
3. 没有候选时，用户看到的是“没有东西”，不是明确的下一步。
4. 同事第一次失败后容易重复刷新，导致网络候选更难保留。
5. 现场排查依赖技术人员口头询问，成本太高。

### 4.3 建议改动

#### 4.3.1 增加“重置并监听当前页”

在监听页主按钮旁新增一个清晰动作：

```text
重新监听当前页
```

行为：

1. 清空当前 `listeningTabId`、`tabNetworkCandidates`、`globalState.candidates`。
2. 重新绑定当前 active tab。
3. 重新注入页面观察脚本。
4. 显示“请回到网页点击播放录音”。

不要只复用普通刷新页面。

#### 4.3.2 增加空状态倒计时提示

监听启动后，如果 20 秒仍没有候选，显示：

```text
还没有发现录音。请确认：
1. 当前打开的是录音详情页。
2. 已点击网页里的播放按钮。
3. 录音正在播放或刚刚发起加载。
4. 如果仍没有出现，请点“复制监听诊断”发给管理员。
```

这个提示不要挡住手动上传入口。

#### 4.3.3 增强扩展诊断

`background.js` 的 `getDiagnostics()` 建议增加：

1. `listeningTabUrl`
2. `listeningTabTitle`
3. `activeTabUrl`
4. `activeTabTitle`
5. `networkCandidateCountForTab`
6. `lastCandidateAt`
7. `lastCandidateHost`
8. `lastContentScriptError`
9. `lastScanAction`
10. `manifestHostPermissions`

`sidepanel.js` 的前端诊断建议增加：

1. 当前 view
2. 是否登录
3. 后端地址
4. 当前候选数量
5. 最近一次候选查重结果
6. 当前插件版本

#### 4.3.4 保留失败后的手动上传兜底

无论监听是否成功，都要保证：

1. 手动上传入口一直可见。
2. 从候选读取失败进入手动上传时，标题和模板选择要带过去。
3. 手动上传成功后仍进入同一套转写和总结流程。

### 4.4 测试要求

新增或更新：

1. `server/background-capture.test.js`
   - 重置监听会清空旧候选。
   - tab 刷新后旧候选不会污染新页面。
   - 诊断包含监听 tab、候选数量、最近错误。
2. `server/sidepanel-ui.test.js`
   - 监听空状态显示操作提示。
   - 复制监听诊断按钮可用。
   - 读取失败后手动上传入口可用。
   - 首页、历史、详情不会被后台候选强行跳回监听页。

人工 E2E：

1. 正常 Plaud 录音：监听、点击播放、出现候选、上传成功。
2. 刷新 Plaud 页面后：点“重新监听当前页”，再次出现候选。
3. 断开后端：候选仍能出现，但查重/上传显示后端失败。
4. 换一条新录音：不会继续显示上一条录音为主候选。

## 5. P2 分享功能设计

### 5.1 用户故事

员工在录音详情页点击“分享”，弹出分享面板：

```text
分享链接

[x] 录音
[x] 逐字稿
[x] 总结

有效期：7 天

复制链接
```

规则：

1. 未生成逐字稿时，“逐字稿”不可勾选。
2. 总结失败或不可用时，“总结”不可勾选，或勾选后提示“当前总结未成功，需先重新生成”。
3. 至少勾选一项才能创建链接。
4. 默认有效期 7 天。
5. 创建成功后复制链接。
6. 已创建的链接可查看、复制、撤销。

### 5.2 权限规则

第一期建议：

1. 记录所有者可以分享自己的记录。
2. 部门负责人可以分享本部门可见记录。
3. 管理员/老板可以分享所有可见记录。
4. 普通员工不能分享别人的记录。
5. 被分享人不需要登录，但只能看到分享人勾选的内容。
6. 分享页面不暴露记录所有者的内部权限信息，不暴露 R2 key、模型错误、后台配置。

如果管理上更保守，可以第一期只允许“记录所有者”和“管理员”创建分享，把部门负责人分享作为二期。

### 5.3 数据表

在 `server/schema/supabase.sql` 新增：

```sql
create table if not exists record_share_links (
  id uuid primary key default gen_random_uuid(),
  audio_record_id uuid not null references audio_records(id),
  created_by uuid not null references employees(id),
  token_hash text not null unique,
  include_audio boolean not null default false,
  include_transcript boolean not null default false,
  include_summary boolean not null default false,
  include_overview_card boolean not null default false,
  include_mind_map boolean not null default false,
  allow_download boolean not null default false,
  expires_at timestamptz not null,
  revoked_at timestamptz,
  access_count integer not null default 0,
  last_accessed_at timestamptz,
  title_override text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists record_share_access_logs (
  id uuid primary key default gen_random_uuid(),
  share_link_id uuid not null references record_share_links(id),
  event_type text not null,
  ip_hash text,
  user_agent text,
  created_at timestamptz not null default now()
);

create index if not exists idx_record_share_links_record on record_share_links(audio_record_id);
create index if not exists idx_record_share_links_token_hash on record_share_links(token_hash);
create index if not exists idx_record_share_links_expires_at on record_share_links(expires_at);
```

同步更新 `server/lib/store.js`：

1. JSON 存储初始化新增 `record_share_links`、`record_share_access_logs`。
2. 旧 `data/dev-db.json` 自动补空表。
3. 不迁移或修改现有录音、逐字稿、总结数据。

### 5.4 Token 设计

不要把登录 JWT 当分享 token。

建议：

1. 创建时生成 32 字节随机值。
2. URL 中使用 base64url token。
3. 数据库只保存 `sha256(token)`。
4. 分享链接格式：

```text
http://<office-host>:8127/s/<token>
```

撤销、过期、删除记录后：

1. `/s/<token>` 返回“链接已失效”页面。
2. `/api/shared/<token>` 返回 404 或 410。
3. 音频接口也必须同步失效。

### 5.5 后端 API

需要新增认证接口：

#### 创建分享

```text
POST /api/records/:id/share-links
```

请求体：

```json
{
  "includeAudio": true,
  "includeTranscript": true,
  "includeSummary": true,
  "includeOverviewCard": false,
  "includeMindMap": false,
  "allowDownload": false,
  "expiresInDays": 7,
  "titleOverride": ""
}
```

校验：

1. `ensureCanViewRecord(auth.employee, record, store)`。
2. 至少勾选一项。
3. 勾选录音时，记录必须有 `original_file_name`。
4. 勾选逐字稿时，必须有 transcript 内容。
5. 勾选总结时，必须有可用总结，不能是 `fallback_template` 或 `invalid`。
6. `expiresInDays` 第一版限制 1 到 30 天。

响应：

```json
{
  "share": {
    "id": "xxx",
    "url": "http://office-host:8127/s/xxxx",
    "expiresAt": "2026-07-01T00:00:00.000Z",
    "includeAudio": true,
    "includeTranscript": true,
    "includeSummary": true
  }
}
```

#### 查看某条记录的分享链接

```text
GET /api/records/:id/share-links
```

返回未过期和已撤销状态，供详情页管理。

#### 撤销分享

```text
DELETE /api/share-links/:shareId
```

或：

```text
PATCH /api/share-links/:shareId
```

请求体：

```json
{ "revoked": true }
```

撤销需要确认创建人或管理员权限。

### 5.6 公开只读接口

这些接口不走登录鉴权，只走分享 token：

#### 分享页面

```text
GET /s/:token
```

返回手机友好的 HTML 页面。

建议新增独立文件：

1. `share.html`
2. `share.css`
3. `share.js`

并在 `server/app.js` 的 `WEB_ASSETS` / 路由中支持。

不要直接复用完整 `sidepanel.js` 登录工作台。分享页应该更轻，且不能带后台管理入口。

#### 分享数据

```text
GET /api/shared/:token
```

返回字段只包含被勾选内容：

```json
{
  "title": "录音标题",
  "createdAt": "2026-06-24T00:00:00.000Z",
  "expiresAt": "2026-07-01T00:00:00.000Z",
  "includes": {
    "audio": true,
    "transcript": true,
    "summary": true
  },
  "audio": {
    "url": "/api/shared/:token/audio",
    "mimeType": "audio/mpeg",
    "fileSize": 12345678,
    "durationSeconds": 3600
  },
  "transcript": {
    "text": "...",
    "segments": []
  },
  "summary": {
    "markdown": "...",
    "overviewCard": {},
    "mindMap": {}
  }
}
```

注意：

1. 未勾选的内容不要返回空字段，直接不返回或返回 `null`。
2. 不返回 `owner_employee_id`、`r2_key`、`model_error`、`provider_errors_json`。
3. 逐字稿 segments 可返回，但要确认不包含后台内部字段。

#### 录音播放

```text
GET /api/shared/:token/audio
```

要求：

1. 只有 `include_audio = true` 时可访问。
2. 支持 `Range` 请求。
3. 支持手机 Safari / Chrome 拖动进度条。
4. 本地文件和 R2 文件都要走同一权限校验。
5. 过期或撤销后立即不可播放。

可复用现有音频服务逻辑，但鉴权从 `authenticate()` 改为 `authenticateShareToken()`。

### 5.7 分享页 UI

分享页首屏建议：

1. 标题。
2. 有效期提示。
3. 如果包含录音，显示原生 `<audio controls>`。
4. 如果包含总结，显示总结卡片和 Markdown 正文。
5. 如果包含逐字稿，显示可滚动逐字稿。

手机体验要求：

1. 375px 宽度可用。

5. 页面无需登录，不出现后台导航、配置、员工管理、上传按钮。
6. 过期页面要说人话：`分享链接已过期，请联系分享人重新生成。`

### 5.8 插件详情页 UI

在 `sidepanel.js` 的详情页工具区新增：

```text
分享
```

弹窗状态：

1. 加载中：读取现有分享列表。
2. 可创建：勾选内容、有效期、复制链接。

### 5.9 下载能力边界

分享第一期建议只做“在线看、在线听”，下载开关默认关闭。

如果要支持下载：

1. `allow_download = true` 才显示下载按钮。
2. 下载接口仍走分享 token。
3. 下载文件名要脱敏，不带内部员工名。
4. 下载也写 `record_share_access_logs`。

## 6. P3 手机和办公室网络适配

### 6.1 publicBaseUrl 必须可被手机访问

后台当前默认：

```text
http://lixindemac-studio.local:8127
```

手机测试要覆盖：

1. iPhone Safari。
2. Android Chrome，若办公室有安卓手机。
3. 同一 Wi-Fi 下访问 `/health`。
4. 同一 Wi-Fi 下访问 `/s/:token`。
5. 关闭 Wi-Fi 后访问失败，这是第一期允许的。

如果 `.local` 不稳定：

1. 给运行后端的 Mac 固定局域网 IP。
2. 管理员配置页把 `publicBaseUrl` 改成 IP 地址。
3. 生成分享链接和下载链接都使用该地址。

### 6.2 Mac 防火墙和端口

必须确认：

1. 运行后端的 Mac 允许局域网访问 8127。
2. 公司 Wi-Fi 没有开启客户端隔离。
3. 同事电脑和手机在同一网段。
4. 浏览器访问不是 HTTPS 时，手机仍允许播放音频。

如果公司 Wi-Fi 开启客户端隔离，局域网分享会失败。此时不能靠代码解决，需要网络侧放行或改成内网服务器部署。

### 6.3 音频播放要求

分享页音频必须：

1. 返回正确 `Content-Type`。
2. 支持 `Accept-Ranges: bytes`。
3. 对 `Range: bytes=...` 返回 206。
4. 不把整段大录音一次性读进内存。
5. R2 临时链接过期时能重新生成或由后端代理读取。

已有 `/api/records/:id/audio` 的 Range 行为可作为参考，但分享接口必须单独校验分享 token。

## 9. 安全和隐私要求

1. 分享 token 是访问凭证，不能太短，不能可枚举。
2. 数据库只保存 token hash。
3. 分享页不能暴露登录 token。
4. 分享页不能暴露后台接口地址以外的内部路径。
5. 分享页不能暴露 R2 key、`r2_key`、模型错误、provider 配置。
6. 过期、撤销、记录删除后，分享页和音频都必须失效。
7. 分享访问日志不保存明文敏感信息。IP 如需记录，保存 hash。
8. 管理员应能看到哪些记录被分享过、谁创建、何时过期、是否撤销。
9. 导出文件和分享链接是两套权限，不要混用 5 分钟下载 token 当 7 天分享链接。
